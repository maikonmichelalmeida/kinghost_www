Raiz/04 RTL Design Synthesis/Timing Analysis

Timing Analysis e Ajuste de SDC

Esta aula transforma os reports de timing em ação prática: como ler um caminho, como decidir se o problema está no SDC, no RTL, na arquitetura, no ambiente de I/O ou apenas na estratégia de compile.

Objetivo desta aula

Depois da síntese, o report de timing é a conversa mais honesta que a ferramenta tem com você. Ele mostra se o circuito consegue respeitar o clock e quais caminhos estão consumindo tempo demais. A dificuldade é que ele parece assustador no começo: muitas colunas, nomes longos, células de biblioteca, increments, arrival, required e slack.

A meta aqui é criar o reflexo correto: ler primeiro o tipo de caminho, depois a conta de timing, depois a causa provável e só então escolher a correção. Um erro comum é usar set_false_path ou set_multicycle_path cedo demais para "limpar" report. Isso pode esconder um bug real. Timing analysis bem feita separa o que precisa ser otimizado do que realmente não deve ser analisado.

Onde esta aula entra no fluxo

O fluxo de síntese não é linear no sentido ingênuo. Você não escreve SDC uma vez, roda compile uma vez e aceita o resultado. O ciclo real é iterativo.

1. SDC inicial

Clock, I/O delay, load, transition e incertezas plausíveis.

2. Compile

DC_NXT tenta mapear e otimizar com base nessas regras.

3. Reports

Timing, constraints, QoR, área e potência mostram o estado.

4. Diagnóstico

Separar violação real, constraint errada e exceção legítima.

5. Ajuste

Corrigir SDC, RTL, path groups ou estratégia de compile.

6. Recompile

Rodar novamente e confirmar que a melhora é real.

Regra de ouro: timing analysis não é procurar um comando que faça o report ficar verde. É descobrir se o design, as constraints e o modelo de ambiente estão contando a mesma história.

Static Timing Analysis em linguagem humana

Static Timing Analysis, ou STA, calcula se sinais conseguem atravessar caminhos combinacionais entre eventos de clock. Ela não simula todos os valores possíveis como um testbench. Em vez disso, analisa atrasos máximos e mínimos em caminhos estruturais.

Clock lança dado
->
Clock-to-Q
->
Lógica + nets
->
Clock captura dado
data arrival time = clock launch + clock-to-Q + delay combinacional + delay de net
data required time = clock capture + período - setup - uncertainty - margens
slack = required - arrival

Se o slack é positivo, o dado chegou antes do prazo. Se é negativo, chegou tarde. A ferramenta pode tentar diminuir atraso escolhendo células mais rápidas, duplicando lógica, inserindo buffers, mudando hierarquia ou reestruturando datapaths. Mas se o prazo foi definido errado no SDC, a ferramenta estará otimizando um problema mal descrito.

Setup e hold: os dois lados da análise

Em curso de RTL, muitas vezes falamos mais de setup porque é intuitivo: o dado precisa chegar antes da borda de captura. Mas hold também importa: logo depois da borda de captura, o dado antigo precisa permanecer estável por um tempo mínimo.

Setup

Verifica se o caminho é rápido o suficiente para entregar o dado até a próxima borda relevante. Violação de setup costuma pedir redução de atraso, relaxamento legítimo de constraint ou pipeline.

Hold

Verifica se o caminho não é rápido demais logo após a borda de captura. Violação de hold costuma pedir buffers, ajuste de skew ou correção de exceção mal aplicada.

Armadilha

Ao aplicar multicycle para setup, você quase sempre precisa ajustar hold. Relaxar setup sem pensar no hold é uma fonte clássica de constraint perigosa.

Como ler um timing report em camadas

Não comece olhando todas as linhas de célula. Primeiro entenda a história geral do caminho. Depois aproxime o zoom.

Camada 1: que tipo de path é este?

Veja Startpoint, Endpoint, Path Group, Path Type e clock de lançamento/captura. Isso diz se o caminho é reg-to-reg, input-to-reg, reg-to-output, input-to-output, setup ou hold.

Camada 2: a conta fecha?

Procure data arrival time, data required time e slack. Se o required parece estranho, talvez o SDC esteja errado. Se o arrival é enorme, talvez o RTL ou o caminho físico esteja pesado.

Camada 3: onde o tempo foi gasto?

Na seção de path delay, compare Incr e Path. Incr é o atraso incremental daquela etapa. Path é o acumulado. Valores grandes em nets podem indicar fanout ou carga; valores grandes em células podem indicar lógica pesada.

Camada 4: o padrão se repete?

Um único caminho falhando pode ser um problema local. Muitos caminhos semelhantes falhando indicam padrão: input delay mal modelado, clock agressivo, path group sem prioridade, interface mal especificada ou arquitetura insuficiente.

Comandos essenciais para investigar timing

O Job Aid do curso mostra muitas opções. Para começar bem, monte uma pequena rotina de investigação. 

# Triagem geral
report_constraint -all_violators
report_qor

# Pior caminho detalhado
report_timing -max_paths 1 -nworst 1

# Mais caminhos para ver padrão
report_timing -max_paths 20 -nworst 5

# Mais detalhes físicos/eléctricos
report_timing -max_paths 10 -nets -capacitance -transition

# Ver grupos de caminho
report_path_group

# Conferir clocks e portas
report_clock
report_port -verbose
ComandoResposta que ele dáQuando usar

max_paths versus nworst

Os slides chamam atenção para uma diferença que confunde bastante. max_paths controla o total de caminhos reportados. nworst controla quantos caminhos piores são reportados por endpoint.

-max_paths

Limita o número total de caminhos. Se você pede -max_paths 10, quer no máximo dez caminhos no report. 

report_timing -max_paths 10

-nworst

Limita quantos caminhos por endpoint aparecem. É útil quando um endpoint sozinho domina o report e esconde outros endpoints problemáticos. 

report_timing -max_paths 20 -nworst 3
Uso prático: se o report sempre mostra variações do mesmo endpoint, use -nworst para enxergar se existem outros endpoints também perigosos.

WNS e TNS: diagnóstico rápido

WNS e TNS não dizem a causa do problema, mas dizem o formato do problema.

SituaçãoLeitura provávelPrimeira ação
WNS pequeno negativo, TNS pequeno Poucos caminhos quase fechando. Investigar path crítico, path group e incremental compile.
WNS muito negativo, TNS pequeno Um caminho estruturalmente ruim. Olhar RTL/datapath; talvez pipeline ou reescrita.
WNS negativo moderado, TNS enorme Muitos caminhos falhando. Suspeitar de SDC, clock agressivo ou ambiente mal modelado.
WNS positivo, DRC violando Timing fecha, mas transição/capacitância/fanout não. Olhar max transition, max capacitance, loads e fanout.

Path groups: orientar a otimização

O DC_NXT otimiza tentando reduzir custo. Se todos os caminhos competem do mesmo jeito, a ferramenta pode gastar esforço onde você não quer. Path groups deixam você dizer: "este conjunto de caminhos merece atenção". 

group_path -name REG2REG \
  -from [all_registers] \
  -to   [all_registers]

group_path -name INPUTS \
  -from [all_inputs] \
  -to   [all_registers]

group_path -name OUTPUTS \
  -from [all_registers] \
  -to   [all_outputs]

group_path -name REG2REG \
  -critical_range 0.3 \
  -weight 2

-critical_range inclui caminhos próximos do pior caminho, para evitar que a ferramenta corrija um caminho e deixe vários quase críticos para trás. -weight dá prioridade relativa. Isso deve ser usado com cuidado: aumentar peso de um grupo pode melhorar esse grupo e piorar outros.

Decidir a correção: SDC, RTL ou compile?

O passo mais importante não é saber comandos. É escolher a categoria correta da correção.

Corrigir SDC Use quando clocks, I/O delays, loads, transitions ou clock groups estão modelados de forma errada.
Corrigir RTL Use quando o caminho é realmente longo: datapath grande, mux profundo, comparador enorme, fanout estrutural.
Ajustar compile Use quando a intenção está correta, mas a ferramenta precisa de foco: path groups, effort, incremental, ungroup.
Adicionar exception Use só quando existe justificativa funcional: caminho falso real, multicycle real, clocks assíncronos reais.
Como saber se é problema de SDC?

Desconfie do SDC quando o report mostra caminhos que não deveriam existir, quando portas aparecem sem delay de entrada/saída, quando clocks virtuais não foram usados para interfaces externas ou quando todos os inputs/outputs parecem críticos de forma artificial. 

report_clock
report_port -verbose
check_timing
report_timing_requirements -ignored
Como saber se é problema de RTL?

Desconfie do RTL quando o mesmo tipo de caminho aparece longo mesmo com constraints plausíveis: cadeia combinacional profunda, operação aritmética larga, mux em cascata, lógica de controle misturada com datapath ou registradores faltando entre estágios naturalmente longos.

Nessa situação, aumentar esforço de compile pode ajudar um pouco, mas não substitui arquitetura. Às vezes a solução real é pipeline, particionamento, redução de fanout ou simplificação de expressão.

Exceptions: remédio forte, não maquiagem

Timing exceptions dizem à ferramenta que certos caminhos não seguem a regra padrão de um ciclo. Elas são necessárias em designs reais, mas perigosas quando usadas para calar reports.

set_false_path

Remove caminho da análise. Use quando funcionalmente o dado nunca precisa ser capturado daquela forma.

set_multicycle_path

Diz que um caminho tem mais de um ciclo para propagar. Precisa tratar setup e hold corretamente.

set_clock_groups

Define relações entre domínios de clock: assíncronos, logicamente exclusivos ou fisicamente exclusivos. 

# Exemplo: clocks assíncronos reais
set_clock_groups -asynchronous \
  -group [get_clocks clk_a] \
  -group [get_clocks clk_b]

# Exemplo: multicycle de setup com hold ajustado
set_multicycle_path 3 -setup -from [get_cells U_SRC/*] -to [get_cells U_DST/*]
set_multicycle_path 2 -hold  -from [get_cells U_SRC/*] -to [get_cells U_DST/*]

# Sempre verificar exceções ignoradas
report_timing_requirements -ignored
Importante para prova e laboratório: uma exception inválida pode não proteger nada. Uma exception ampla demais pode esconder um bug. Por isso, depois de qualquer exception, rode report_timing_requirements -ignored e faça um report filtrado no caminho que você queria afetar.

Multicycle: por que hold costuma ser setup - 1

Um multicycle path existe quando a lógica foi projetada para levar mais de um ciclo entre launch e capture. O comando de setup muda a borda de captura usada na análise. Só que a análise de hold também muda de forma implícita se você não corrigir. Por isso aparece a regra prática: se setup é relaxado para N ciclos, hold normalmente precisa ser ajustado para N-1.

Default

Setup captura na próxima borda. Hold verifica borda muito próxima.

Setup N

Captura é empurrada para uma borda mais distante.

Hold N-1

Recoloca a checagem de hold no ponto correto.

set_multicycle_path 4 -setup -from [get_clocks src_clk] -to [get_clocks dst_clk]
set_multicycle_path 3 -hold  -from [get_clocks src_clk] -to [get_clocks dst_clk]

Não aplique isso por fórmula cega. Antes, confirme que existe uma razão funcional: enable, protocolo, FSM, handshake ou arquitetura que realmente só captura depois de vários ciclos.

Quando o problema está nos I/Os

Muitos problemas de timing em blocos pequenos não estão no reg-to-reg interno, mas nas interfaces. O SDC de I/O é uma forma de dizer quanto tempo o mundo externo já gastou antes de entregar uma entrada, ou quanto tempo o mundo externo precisa depois de receber uma saída.

Entrada

set_input_delay representa o tempo consumido pelo bloco anterior antes de o sinal chegar ao nosso input. Quanto maior o input delay, menor o tempo interno disponível. 

set_input_delay -clock master_clk 0.8 \
  [remove_from_collection [all_inputs] [get_ports clk]]

Saída

set_output_delay representa o tempo que o bloco externo precisa depois da nossa saída. Quanto maior o output delay, menor o tempo que nosso bloco pode gastar até a saída. 

set_output_delay -clock master_clk 0.8 [all_outputs]

Se o report mostra muitos caminhos de input ou output críticos, revise se os delays são realistas. Não é errado começar com valores simplificados no laboratório, mas é importante saber que eles são hipóteses.

Clock virtual: quando a interface não usa o clock interno diretamente

Um clock virtual é um clock sem porta física no design. Ele representa o clock do dispositivo externo. Isso é muito útil quando o bloco se comunica com outro domínio ou com uma interface cujo clock de referência não deve ser tratado como o clock interno. 

create_clock -name ext_clk -period 20.0

set_input_delay  -clock ext_clk 3.0 [get_ports data_in]
set_output_delay -clock ext_clk 2.0 [get_ports data_out]

No começo do nosso laboratório, o bloco CPU provavelmente pode usar um clock simples. Mas quando você começar a justificar interfaces mais realistas, clocks virtuais serão uma forma mais correta de modelar o ambiente externo sem bagunçar o clock interno.

Fluxo recomendado para o nosso projeto agora

Para o que o professor pediu até aqui, a prática mais útil é criar uma rotina curta: rodar síntese, ler reports, ajustar SDC de forma pequena e repetir. Não precisamos ainda fazer Formality; mas já precisamos produzir uma base que não atrapalhe Formality depois. 

cd PROJECT_ENVIROMENT/module_CPU_pdk_SAED/blocks/CPU

make synth

# Depois da síntese, olhar:

Aplicação direta no seu bloco: o primeiro SDC não precisa ser perfeito. Ele precisa ser explícito, auditável e melhor que "sem constraints". Depois dos reports, a gente melhora com base em evidência.

Erros comuns de iniciante

ErroPor que é perigosoCorreção mental
Usar false path para sumir com violação Pode esconder caminho funcional real. Justifique funcionalmente antes de aplicar.
Não ajustar hold em multicycle Setup relaxa, mas hold pode ficar incorreto. Para setup N, revise hold N-1.
Confiar só em report_timing Você pode ignorar DRCs e violações agregadas. Use também report_constraint e report_qor.
Não modelar I/O O bloco fica otimizado como se o mundo externo fosse ideal. Use input/output delay, transition e load.
Mudar várias coisas por rodada Você perde a causa da melhora ou piora. Faça uma mudança pequena, rode e compare reports.

Perguntas de revisão

1. O que significa slack negativo?

O dado chegou depois do tempo requerido; o caminho viola timing para aquela análise.

2. Por que report_constraint e report_timing são complementares?

O primeiro mostra o quadro geral de violações; o segundo explica caminhos específicos em detalhe.

3. Quando usar set_false_path?

Quando há justificativa funcional de que aquele caminho não deve ser analisado como caminho temporal real.

4. Qual é a armadilha do multicycle?

Relaxar setup sem corrigir hold pode criar uma constraint temporal errada.

5. O que -nworst ajuda a enxergar?

Ajuda a limitar caminhos por endpoint, evitando que um único endpoint esconda outros problemas.

6. Por que clock virtual é útil?

Ele modela o clock de um bloco externo sem criar uma porta de clock física dentro do design.
