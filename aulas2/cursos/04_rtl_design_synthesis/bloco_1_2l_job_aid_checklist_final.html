Job Aid DC_NXT: Checklist Final

Esta é a página de fechamento do curso: um mapa operacional para reconhecer comandos, revisar prova e transformar o conteúdo em uma sequência de trabalho no laboratório do professor.

Objetivo desta aula

O Job Aid não é uma aula conceitual comum. Ele é uma cola operacional: como invocar a ferramenta, configurar bibliotecas, usar Tcl, buscar objetos, aplicar constraints, gerar reports, compilar, lidar com SPG e escrever arquivos de saída. Esta página resume o que você deve carregar na cabeça quando abrir um Makefile ou script do professor.

A ordem do fluxo importa

1. SetupBibliotecas, search path, work library, SVF e variáveis.
2. Read RTLanalyze, elaborate, link.
3. ConstraintsSDC, clocks, I/O, ambiente, exceptions e checks.
4. Compilecompile_ultra, opções, path groups, scan, SPG.
5. ReportsTiming, área, power, QoR, constraints, congestion.
6. OutputNetlist, DDC, SDC, ICC2 files e artefatos para Formality.

Invocação e ajuda

# Modo topographical / DC NXT
dcnxt_shell -topo

No servidor da UFRGS, você provavelmente não vai decorar a invocação completa se o Makefile do professor já encapsular isso. Mas precisa reconhecer se o target está abrindo a ferramenta correta e se o modo topographical está ativo quando o fluxo exigir recursos físicos.

Tcl e collections

O DC NXT não trabalha só com strings. Muitos comandos retornam collections: conjuntos de portas, células, nets, clocks ou designs. Entender isso evita scripts frágeis. 

set regs [all_registers]
sizeof_collection $regs

get_ports clk
get_clocks CORE_CLK
get_cells -hier *alu*
get_nets -hier *result*
Regra prática: sempre que possível, passe objetos retornados por get_*, não nomes digitados soltos. Isso reduz erro de typo e melhora scripts reutilizáveis.

Bloco de constraints essencial

create_clock -name CORE_CLK -period 20.0 [get_ports clk]
set_clock_uncertainty 0.2 [get_clocks CORE_CLK]

set_input_delay  2.0 -clock CORE_CLK [remove_from_collection [all_inputs] [get_ports clk]]
set_output_delay 2.0 -clock CORE_CLK [all_outputs]

set_input_transition 0.1 [remove_from_collection [all_inputs] [get_ports clk]]
set_load 0.05 [all_outputs]

check_timing
report_clocks
report_timing_requirements -ignored

Este modelo não deve ser copiado sem pensar, mas é uma base mental. Para o nosso CPU, precisamos adaptar período, delays, cargas, reset e portas reais. A qualidade está em documentar hipóteses e validar o que a ferramenta entendeu.

Compile dirigido

Opção/recurso Uso Cuidado
compile_ultra Síntese otimizada principal. Não substitui constraints corretas.
-scan Prepara síntese para scan/DFT. Use quando o fluxo exigir scan-ready.
-retime Permite mover registradores para melhorar timing. Exige Formality/SVF bem configurado.
-spg Usa Synopsys Physical Guidance. Faz sentido com setup físico/floorplan adequado.
-incremental Ajusta solução existente. Não é ideal para mudanças grandes.

SPG, NLO e congestionamento no Job Aid

O Job Aid fecha o arco do curso: síntese lógica, constraints, compile ultra e depois síntese com consciência física. SPG não é o primeiro passo do nosso CPU, mas é o vocabulário para entender o sistema do professor quando ele aproxima DC NXT de ICC II. 

compile_ultra -spg
report_congestion -build_map
report_congestion
report_net_routing_layer_constraints [get_nets -hier] -output net_layer.tcl

Checklist final do curso

Antes de ler RTL Setup carregado, bibliotecas resolvidas, diretórios existem, SVF definido se haverá Formality.
Depois de ler RTL current_design correto, link sem referência quebrada, check_design lido.
Depois do SDC Clocks, I/O delays, loads, transitions e exceptions conferidos com reports.
Depois do compile QoR, timing, área, power e constraints analisados. Não basta ver "rodou".
Antes de entregar Netlist limpa, artefatos salvos, logs organizados e hipóteses documentadas.

Mini revisão de prova

Tema Resposta curta que precisa estar na cabeça
GTECH Representação genérica/intermediária antes de mapear para target library.
Target library Biblioteca de células disponíveis para implementação.
Link library Bibliotecas usadas para resolver referências no design.
Timing path Caminho entre startpoint e endpoint: reg-to-reg, input-to-reg, reg-to-output ou combinacional.
DRC Design rule constraints, como max transition, max capacitance e fanout; costumam ter prioridade alta.
Physical synthesis Síntese que considera efeitos físicos, floorplan, RC e posicionamento aproximado.

Fechamento: o que fazer agora no laboratório

Com este curso fechado, o caminho mais produtivo é aplicar o conhecimento no projeto, não continuar acumulando teoria. O próximo passo real é pegar o esqueleto do professor, encaixar o CPU em um bloco, escrever/ajustar SDC, rodar simulação e síntese por Makefile, guardar reports e preparar a ponte para Formality.

Direção recomendada: fazer o mínimo completo primeiro. Um fluxo pequeno que simula, sintetiza, escreve netlist e gera reports vale mais do que um fluxo enorme com SPG, retiming e Formality ainda sem base.
