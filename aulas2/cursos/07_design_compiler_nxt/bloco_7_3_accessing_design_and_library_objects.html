Accessing Design and Library Objects

Esta aula ensina a enxergar o DC NXT como um banco de dados de objetos: designs, cells, ports, pins, nets, clocks, bibliotecas, células de biblioteca, atributos e collections.

Objetivo da aula

Esta aula é uma das mais importantes para sair do modo "decorar comando" e entrar no modo "entender o que o script está fazendo". No Design Compiler NXT, comandos de síntese, constraints, reports e debug não trabalham apenas com nomes escritos como texto. Eles trabalham com objetos do banco de dados interno.

Quando você escreve set_input_delay, set_load, report_timing, set_dont_touch ou compile_ultra, a ferramenta precisa saber em quais objetos agir. Se você seleciona o objeto errado, a constraint pode não pegar, pode pegar em outro lugar ou pode retornar vazio sem você perceber.

Modelo mental: RTL vira banco de dados

O RTL é texto. Depois de analyze e elaborate, o DC NXT transforma esse texto em objetos manipuláveis. O que no Verilog parece módulo, instância, fio e porta passa a existir como objetos consultáveis por Tcl. 

module TOP (A, B, C, D, CLK, OUT1);
  input        A, B, C, D, CLK;
  output [1:0] OUT1;

  wire bus0, bus1, INV0, INV1;

  ENCODER U1 (.AIN(A), .Q1(bus1));
  INVX1   U2 (.A(bus0), .Y(INV0));
  INVX1   U3 (.A(bus1), .Y(INV1));
  REGFILE U4 (.D0(INV0), .D1(INV1), .CLK(CLK));
endmodule

Depois da elaboração, TOP não é apenas uma palavra no arquivo. Ele é um objeto design. As instâncias U1, U2, U3 e U4 são objetos cell. Os fios bus0, bus1, INV0 e INV1 são objetos net. As entradas e saídas do módulo são ports.

Frase para guardar: em scripts Synopsys, você quase nunca quer "um nome qualquer". Você quer uma collection de objetos do tipo certo.

Os principais objetos do DC NXT

A primeira habilidade da aula é classificar corretamente o que você está selecionando. Isso parece simples, mas é a raiz de muitos erros de SDC.

Objeto O que representa Exemplo Comando típico
design Módulo/entity elaborada na memória da ferramenta. TOP, ENCODER, REGFILE get_designs *
cell Instância dentro do design atual. U1, U2, I_REG get_cells *
port Entrada ou saída do design atual. A, CLK, OUT1[0] get_ports *
pin Terminal de uma instância dentro do design atual. U1/AIN, U4/CLK get_pins */CLK
net Conexão entre ports e pins. bus0, SUM, INV1 get_nets *
clock Objeto criado por constraint de clock. CLK, depois de create_clock get_clocks *
lib_cell Célula disponível na biblioteca tecnológica. 20nm/INV1, 20nm/OR2_4x get_lib_cells 20nm/INV*
lib_pin Pin definido na célula de biblioteca. 20nm/OR2_4x/Y get_lib_pins 20nm/OR2_4x/*

design, cell e lib_cell

Esta é uma distinção que precisa ficar muito bem assentada. ENCODER pode ser um design porque é um módulo lido e elaborado. I_ENC é uma cell porque é uma instância de ENCODER dentro de outro design. INVX1 é uma lib_cell porque vem da biblioteca tecnológica, não do RTL do usuário.

get_designs Consulta blocos elaborados, como TOP, ENCODER e REGFILE.
get_cells Consulta instâncias dentro do design, como U1, U2 e I_REG.
get_lib_cells Consulta células da biblioteca, como INV1, NAND2, DFF. 
get_designs *
# {TOP ENCODER REGFILE PLL}

get_cells *
# {I_ENC U1 U2 I_REG}

get_lib_cells 20nm/INV*
# {20nm/INV1 20nm/INV2 20nm/INV3}
Pergunta clássica

Por que INVX1 não é um design como ENCODER? Porque INVX1 é uma célula da biblioteca tecnológica. Ela pode ser a referência de uma instância, mas não foi elaborada como módulo RTL do seu projeto.

Port versus pin

Port e pin não são categorias absolutas. Elas dependem do nível hierárquico que você está olhando. Se o current_design é TOP, a entrada de uma instância interna é um pin. Se você entra no design dessa instância, o mesmo terminal funcional aparece como port. 

No design pai:
  U2/MI  -> pin da instância U2

Dentro do próprio design MID:
  MI     -> port do design MID
Port

Fronteira do design atual

Entradas, saídas e inouts do módulo/entity que está ativo como current_design.

Pin

Terminal de instância

Entradas e saídas das cells instanciadas dentro do design atual.

Por que isso importa em SDC?

Constraints de interface, como set_input_delay e set_output_delay, normalmente são aplicadas em ports do top. Já constraints internas ou debug de timing muitas vezes usam pins de instâncias. Se você usa get_ports quando deveria usar get_pins, pode receber uma collection vazia.

Nomes ambíguos: por que usar get_*

O slide mostra um caso com um port chamado SUM e uma net também chamada SUM. Se você escreve apenas: 

set_load 5 SUM

você deixa a ferramenta resolver a ambiguidade. Em laboratório, isso é perigoso. O jeito robusto é dizer explicitamente qual classe de objeto você quer atingir. 

# Carga no port SUM
set_load 5 [get_ports SUM]

# Carga na net SUM
set_load 5 [get_nets SUM]
Regra prática: em scripts versionados, prefira sempre passar collections explícitas: [get_ports ...], [get_pins ...], [get_cells ...].

Como ler colchetes Tcl

Os colchetes em Tcl executam o comando interno primeiro. Isso é essencial para entender os scripts Synopsys. O comando abaixo não passa o texto get_ports SUM para set_load; ele executa get_ports SUM antes e passa o resultado. 

set_load 5 [get_ports SUM]
1. Executa get_ports SUM.
2. Retorna Uma collection com o port SUM.
3. Aplica set_load 5 nessa collection.

O papel do current_design

Muitos comandos consultam objetos no nível do design atual. Se o current_design é TOP, get_ports * retorna os ports de TOP. Se você muda o current design, o mesmo comando pode retornar outro conjunto. 

current_design TOP
get_ports *
# ports de TOP

current_design ENCODER
get_ports *
# ports de ENCODER
Boa prática: antes de aplicar constraints importantes, garanta explicitamente o top com current_design $TOP. Isso reduz a chance de aplicar SDC no nível errado.

Consultas hierárquicas e flat

A aula mostra uma pegadinha: get_cells -hierarchical e get_flat_cells parecem resolver o mesmo problema, mas não consultam exatamente do mesmo jeito.

Comando Como pensar Exemplo
get_cells sd_mux* Busca no nível atual. Células locais chamadas sd_mux....
get_cells -hierarchical sd_mux* Busca por nome local em toda a hierarquia. Todas as instâncias sd_mux... abaixo do top.
get_flat_cells *I_SDRAM_IF*sd_mux* Busca no nome completo achatado. Caminhos que contêm I_SDRAM_IF e sd_mux. 
get_cells I_SDRAM_TOP/I_SDRAM_IF/sd_mux*

get_cells -hierarchical sd_mux*

get_flat_cells *I_SDRAM_IF*sd_mux*
A pegadinha do / com hierarquia

Em alguns comandos, / funciona como delimitador hierárquico. Em outros contextos, especialmente com -hierarchical, o padrão esperado pode ser o nome local da célula, não o caminho completo. Se uma busca retorna vazio, teste se você deveria usar get_flat_cells ou simplificar o padrão.

-of_objects: navegando pelo banco

A opção -of_objects permite descobrir objetos associados a outros objetos. Ela é muito útil quando você não sabe o nome exato, mas sabe a conexão. 

# Quais cells estão associadas a uma net?
get_cells -of_objects [get_nets net_sdram_clk]

# Quais pins pertencem a uma instância?
get_pins -of_objects [get_cells U1]

# Qual net está conectada a um pin?
get_nets -of_objects [get_pins U1/Q]

Esse tipo de navegação é excelente para debug. Em vez de tentar adivinhar nomes, você caminha pelo grafo do design: cell para pins, pin para net, net para outras cells.

Consultando bibliotecas

O banco de dados não contém apenas o design. Ele também contém bibliotecas carregadas e suas células. Isso permite investigar área, direção de pins e células disponíveis sem abrir manualmente arquivos .lib ou .db. 

get_libs
# {20nm gtech standard.sldb}

get_lib_cells 20nm/OR2_4x

get_lib_pins 20nm/OR2_4x/*

get_lib_attribute 20nm/OR2_4x area

get_lib_attribute 20nm/OR2_4x/Y direction
Aplicação prática: quando uma célula aparece em relatório, você pode consultar sua biblioteca, seus pins e atributos. Isso ajuda a entender por que o compile escolheu certa implementação ou por que uma constraint elétrica foi violada.

Comandos all_*

Os comandos all_inputs, all_outputs, all_clocks e all_registers são atalhos que retornam collections prontas. Eles são muito usados em SDC, mas não aceitam padrões arbitrários como os comandos get_*.

Comando Retorna
all_inputs Todos os input e inout ports do design atual.
all_outputs Todos os output e inout ports do design atual.
all_clocks Todos os clocks definidos no nível atual ou abaixo.
all_registers Todos os registradores na hierarquia inteira do design atual. 
# Válido
all_inputs

# Inválido: all_inputs não espera padrão posicional
all_inputs C*

# Forma comum: todos os inputs exceto o clock
remove_from_collection [all_inputs] [get_ports CLK]
Ponto de prova

all_inputs C* gera erro. Para filtrar, use get_ports C* ou filter_collection [all_inputs] "name =~ C*".

Collections: o coração dos scripts Synopsys

Uma collection é uma referência interna para objetos do banco de dados Synopsys. Ela não é uma lista Tcl comum. O slide mostra que, ao guardar uma collection em uma variável, o valor pode aparecer como um handle interno, por exemplo _sel184. 

set foo [get_ports p*]
echo $foo
# _sel184

query_objects $foo
# {pclk pframe_n pidsel pad[31] ...}

sizeof_collection $foo
# 50
query

Ver o conteúdo

query_objects $collection mostra os nomes dos objetos.

count

Contar objetos

sizeof_collection $collection informa quantos objetos foram encontrados.

combine

Juntar ou remover

add_to_collection junta; remove_from_collection subtrai.

Filtrando por atributos

Objetos têm atributos. Em vez de filtrar só por nome, você pode filtrar por propriedades, como ref_name, dont_touch, is_mapped, is_hierarchical ou period. 

# Cells cuja referência começa com AN
filter_collection [get_cells *] "ref_name =~ AN*"

# Cells ainda não mapeadas
filter_collection [get_cells *] "is_mapped != true"

# Atalho com -filter
get_cells * -filter "dont_touch == true"

# Clocks rápidos
set fastclks [get_clocks * -filter "period < 10"]
Operador Uso
==Igualdade.
!=Diferença.
>, <, >=, <=Comparações numéricas.
=~Casa com padrão.
!~Não casa com padrão.

Iterando: foreach versus foreach_in_collection

Uma lista Tcl e uma collection Synopsys podem parecer parecidas quando impressas, mas são estruturas diferentes. Para lista Tcl, use foreach. Para collection, use foreach_in_collection. 

# Lista Tcl comum
set nomes {A B C D}
foreach nome $nomes {
  puts $nome
}

# Collection Synopsys
set ports [get_ports *]
foreach_in_collection p $ports {
  echo [get_object_name $p]
}

Dentro de foreach_in_collection, a variável p é um handle de objeto. Para imprimir o nome legível, use get_object_name. 

foreach_in_collection cell [get_cells -hier * -filter \
    "is_hierarchical == true"] {
  echo "Instance [get_object_name $cell] is hierarchical"
}

Como isso aparece no SDC

Esta aula prepara diretamente a próxima, que entrará em constraints. Um SDC saudável seleciona objetos com precisão. O exemplo clássico é aplicar input delay em todos os inputs menos o clock. 

create_clock -name CLK -period 10 [get_ports CLK]

set_input_delay 2.0 -clock CLK \
  [remove_from_collection [all_inputs] [get_ports CLK]]

set_output_delay 2.0 -clock CLK [all_outputs]

Agora esse script deixa de ser uma fórmula decorada:

[get_ports CLK] Seleciona o port de clock no top.
[all_inputs] Retorna todos os inputs do design atual.
remove_from_collection Remove o clock da collection de inputs.
[all_outputs] Seleciona todos os outputs para aplicar output delay.

Debug: quando a ferramenta não encontra objetos

Muitos erros de laboratório começam com uma mensagem parecida com: 

Warning: No objects matched 'CLK'
Error: extra positional option 'C*' (CMD-012)

A resposta não deve ser chutar comandos até funcionar. Faça uma investigação curta e ordenada.

1. Confira o design atual current_design e get_designs *.
2. Confira o tipo de objeto Teste get_ports CLK, get_pins */CLK, get_nets CLK.
3. Conte a collection Use sizeof_collection para saber se encontrou algo.
4. Veja os nomes reais Use query_objects ou padrões mais amplos, como get_ports *CLK*.

Aplicação no nosso laboratório

Para o projeto do professor, esta aula é a base de scripts robustos. Quando montarmos ou auditarmos Makefiles e Tcl, vamos usar essa lógica para garantir que constraints e reports estão pegando nos objetos corretos.

Uso prático imediato: quando criarmos o SDC do bloco, não vamos escrever constraints em nomes crus. Vamos selecionar ports, clocks, registers, cells e outputs com comandos get_*, all_* e collections. Isso deixa o script explicável e depurável. 
# Pequeno bloco de sanidade que pode aparecer em script Tcl
puts "TOP atual:"
current_design

puts "Ports encontrados:"
query_objects [get_ports *]

puts "Inputs exceto clock:"
query_objects [remove_from_collection [all_inputs] [get_ports CLK]]

puts "Registradores:"
query_objects [all_registers]

Revisão forte

O que get_* retorna?

Uma collection de objetos do banco de dados Synopsys, não uma lista Tcl comum.

Qual é a diferença entre port e pin?

Port é terminal do design atual. Pin é terminal de uma instância dentro do design atual.

Por que INVX1 não é design?

Porque é célula de biblioteca. A instância que usa INVX1 é uma cell; a definição é uma lib_cell.

Como remover clock dos inputs?

remove_from_collection [all_inputs] [get_ports CLK].

Como iterar sobre collection?

Com foreach_in_collection, usando get_object_name para imprimir nomes legíveis.
