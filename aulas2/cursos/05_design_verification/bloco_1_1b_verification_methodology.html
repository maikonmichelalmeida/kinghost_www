Raiz/05 Design Verification/Methodology

Verification Methodology

Metodologia é o que transforma testes soltos em um processo confiável: objetivo claro, plano de verificação, ambiente reutilizável, checagem automática, cobertura, regressão e critérios de conclusão.

Objetivo desta aula

A aula anterior mostrou que simular não é o mesmo que verificar. Esta aula responde o próximo problema: se não dá para testar tudo, como decidir o que testar, como saber se ainda falta algo e como evitar que a verificação vire uma coleção de testes improvisados?

A resposta do curso é metodologia. Não é burocracia; é engenharia de repetição. Um bom método permite que você rode testes muitas vezes, meça progresso, descubra buracos, adicione casos, preserve os testes antigos e chegue a uma decisão defensável de "por enquanto, este bloco está suficientemente verificado".

A meta não é chegar a 100% de um número

Os slides começam com uma armadilha: cobertura parece uma nota de prova, mas não é. Code coverage alto não significa design correto. Functional coverage alto também não prova ausência de bugs se o plano estiver incompleto. Coverage é uma ferramenta para orientar a investigação, não um certificado automático de qualidade.

Completude

Mede se aquilo que você decidiu observar foi exercitado. É uma pergunta sobre alcance dos testes.

Correção

Mede se o comportamento observado está de acordo com a especificação. É uma pergunta sobre julgamento.

Risco

Um ambiente pode ser completo no que mede e ainda estar errado no que conclui.

Frase para guardar: coverage diz onde você passou; checker diz se o que aconteceu ali estava certo. Um sem o outro é frágil.

Code coverage versus functional coverage

Code coverage olha para o RTL e pergunta: quais linhas, branches, condições, toggles ou estados de FSM foram exercitados? Functional coverage olha para o plano e pergunta: quais cenários funcionais importantes foram exercitados?

Cenário Interpretação correta Próxima ação
100% code coverage, 50% functional coverage O RTL foi executado, mas metade dos cenários funcionais planejados ainda não aconteceu. Criar testes para os cenários faltantes.
50% code coverage, 100% functional coverage O plano funcional foi coberto, mas há código não exercitado; talvez o plano esteja incompleto. Investigar código morto, casos não planejados ou funcionalidades esquecidas.
Alta cobertura, sem checker O design foi muito exercitado, mas pouco julgado. Adicionar self-check, assertions ou scoreboard.
Baixa cobertura, testes passando Os testes atuais são insuficientes para confiança. Ampliar estímulos, regressão e plano.
Por que code coverage baixa é um diagnóstico claro?

Se uma linha, branch, condição ou estado nunca foi exercitado, você não tem evidência dinâmica sobre aquele trecho. Pode ser código morto, caso impossível, feature removida, ou simplesmente falta de teste. Em todos os casos, há uma decisão a tomar: excluir com justificativa, remover o código ou criar teste.

Quando um teste ou bloco está "done"?

O curso resume uma ideia forte: Test Done = Test Plan Executed and All Code Executed. Essa frase não deve ser lida literalmente como "acabou para sempre". Ela significa que, para declarar uma etapa concluída, você precisa ter executado o plano de verificação e também investigado cobertura de código.

Plano executado Todos os cenários funcionais planejados foram rodados e checados.
Código exercitado Linhas, branches, condições, toggles e FSMs relevantes foram cobertos ou justificados.
Falhas resolvidas Bugs encontrados foram corrigidos, e regressão confirma que não voltaram.
Para nosso CPU: "done" não deveria significar "abri a waveform e parece bom". Um teste deve produzir PASS/FAIL, e o conjunto de testes deve cobrir instruções, resets, flags, memória, desvios e casos de erro que decidirmos colocar no plano.

Tecnologias de verificação

O curso coloca as tecnologias em uma escala. Cada uma resolve um tipo de problema. Um simulador event-driven é detalhado e ideal para debug inicial. Um simulador cycle-based ganha desempenho em certos cenários. Emulação e aceleração permitem rodar muito mais ciclos, mas exigem infraestrutura maior. Formal prova propriedades ou equivalência em espaços onde simulação seria insuficiente.

Tecnologia Força Limitação
Event-driven simulation Detalhe temporal, debug de sinais, boa compatibilidade com testbenches ricos. Pode ser lenta em designs grandes e regressões longas.
Cycle-based simulation Mais rápida quando a precisão por evento não é necessária. Menos flexível para certos efeitos e debug fino.
Emulation/acceleration Roda muitos ciclos e aproxima workloads reais. Setup caro e menos confortável para debug detalhado.
Formal/equivalence Prova propriedades ou equivalência sem enumerar testes manualmente. Depende de propriedades/modelagem corretas e pode ter limites de escala.

Estratégia começa pela especificação

Uma metodologia de verificação começa antes de escrever testbench. Primeiro você entende a especificação e decide quais comportamentos precisam ser demonstrados. Depois transforma esses comportamentos em itens de plano: estímulo, observação, checker, cobertura e prioridade.

1. Especificação

Quais modos, entradas, saídas, protocolos, estados, exceções e restrições definem o bloco?

2. Plano de verificação

Quais cenários testam esses requisitos? O que é must-pass? O que é corner case? O que precisa de coverage?

3. Ambiente

Como drivers, monitors, checkers, scoreboard e coverage serão organizados para reaproveitamento?

4. Regressão

Como rodar repetidamente, coletar resultados, comparar logs e evitar que bugs corrigidos voltem?

Coverage Driven Verification como ciclo

Coverage-driven verification não é uma etapa única. É um loop. Você roda testes, olha cobertura, encontra buracos, cria novos testes ou constraints, roda de novo e repete. No começo, cobertura sobe rápido porque os casos básicos exercitam muita coisa. No final, cada ponto restante custa mais: são corner cases, combinações raras e caminhos difíceis.

Preliminary verification Testes simples para garantir que o bloco respira: reset, operação básica, interfaces vivas.
Build environment Criar estrutura reutilizável: driver, monitor, checker, scoreboard, coverage e regressão.
Broad-spectrum Explorar grande faixa de cenários, muitas vezes com randomização controlada.
Corner-case Mirar casos raros, limites, exceções, conflitos e estados pouco prováveis.
Directed closure Escrever testes específicos para fechar buracos restantes de coverage.
Por que o final da cobertura é difícil?

Os primeiros testes atravessam os caminhos comuns. Os últimos pontos de cobertura geralmente representam combinações raras, ordem específica de eventos, estados internos pouco acessíveis ou interações que só acontecem com constraints bem pensadas. Por isso, no final, testes dirigidos continuam importantes mesmo em ambientes com randomização.

Arquitetura de testbench

Os slides mostram uma arquitetura de testbench porque metodologia precisa de estrutura. Se cada teste dirige sinais manualmente, observa manualmente e decide manualmente, o ambiente fica difícil de manter. Separar papéis permite crescer.

Test

Escolhe cenário, configura ambiente e inicia estímulos.

Stimulus generator

Cria transações dirigidas ou randomizadas conforme o plano.

Driver

Converte transações em sinais no protocolo do DUT.

Monitor

Observa sinais e reconstrói transações ou eventos relevantes.

Scoreboard/checker

Compara esperado versus observado e acusa erro automaticamente.

Coverage collector

Mede cenários exercitados e orienta próximos testes.

O coração da metodologia: tests mudam; testbench permanece. Você quer trocar cenário sem reescrever todo o ambiente.

Coverage não substitui self-check

Um erro muito comum é tratar coverage como verificador. Coverage só diz que algo aconteceu. Ela não diz se aconteceu corretamente. Por isso a metodologia exige self-checking testbenches: testes que sabem falhar sozinhos.

Coverage responde

O cenário foi exercitado? O branch executou? A combinação apareceu?

Checker responde

O resultado está correto? O protocolo foi respeitado? A latência foi a esperada?

// Ideia simples de self-check
if (dut_result !== expected_result) begin
  $error("Resultado incorreto: esperado=%0d observado=%0d", expected_result, dut_result);
end

Accellera e padronização

O bloco menciona Accellera porque metodologias de verificação não vivem só dentro de uma empresa. A indústria padroniza linguagens, bibliotecas e metodologias para permitir interoperabilidade, reuso e treinamento. Isso é parte do caminho que leva a SystemVerilog e UVM nos próximos blocos.

Ligação com o próximo bloco: quando o curso falar de UVM, não pense primeiro em macros e sintaxe. Pense no problema que UVM tenta resolver: padronizar arquitetura de testbench, separação de testes, reuso de componentes, comunicação por transações e automação de regressão.

Aplicação no nosso CPU

Para o nosso projeto, o melhor próximo passo não é pular direto para UVM completo. É aplicar metodologia em uma escala pequena e clara. Um CPU simples já permite um bom plano: instruções aritméticas, loads/stores, jumps, flags, reset, memória, casos inválidos e combinações de sequência.

Plano Criar uma tabela de funcionalidades e cenários obrigatórios.
Self-check Cada teste precisa comparar estado final esperado, registradores, memória e flags.
Regressão Makefile deve rodar todos os testes e sair com erro se qualquer teste falhar.
Coverage inicial Começar simples: quais instruções foram exercitadas, quais flags foram observadas, quais branches ocorreram.

Roteiro de implementação incremental

  1. Transformar cada cenário do CPU em teste nomeado.
  2. Adicionar mensagens PASS/FAIL e retorno de erro para integração com Makefile.
  3. Separar estímulo, observação e checagem mesmo que ainda em SystemVerilog simples.
  4. Adicionar um scoreboard simples baseado em resultado esperado.
  5. Adicionar cobertura funcional mínima: instrução, flag, tipo de acesso e casos de reset.
  6. Rodar regressão antes de síntese, depois repetir quando o RTL mudar.

Revisão para fixar

1. Coverage alta prova correção? Não. Coverage mede exercício; correção vem de checkers, assertions, scoreboard e comparação contra especificação.
2. Por que code coverage e functional coverage se complementam? Code coverage olha para o RTL executado; functional coverage olha para os cenários planejados.
3. O que significa test done? Plano executado, código relevante exercitado, falhas resolvidas e buracos justificados ou fechados.
4. Por que regressão é parte da metodologia? Porque bugs corrigidos podem voltar, e mudanças pequenas podem quebrar funcionalidades antigas.
5. Por que testes dirigidos continuam importantes? Porque corner cases específicos muitas vezes não aparecem com randomização ampla sem orientação.
