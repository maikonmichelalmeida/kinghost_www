Raiz/05 Design Verification/Introduction

Introduction to Verification

Verificação é o processo de ganhar confiança, com evidências, de que um design implementa a especificação pretendida. Ela não é apenas “rodar uma simulação”; é planejar estímulos, observar respostas, detectar erros automaticamente e medir o que foi coberto.

Objetivo desta aula

O curso começa com uma mudança mental importante: em RTL, compilar e simular sem erro não significa que o bloco está correto. A simulação só mostra que, para aqueles estímulos, o modelo produziu aquelas respostas. Se o testbench não checa o resultado, se os estímulos não exploram os casos importantes, ou se a especificação foi interpretada errado, o bug passa quieto.

Para o nosso projeto, essa aula é a base para sair de um script que “roda o testbench” e ir em direção a um ambiente de verificação: estímulos controlados, monitoramento, checagem, cobertura e debug com VCS/Verdi.

Por que verificação é necessária?

Em VLSI, corrigir um erro depois da fabricação é extremamente caro. Mesmo em FPGA ou laboratório acadêmico, um bug descoberto tarde contamina síntese, reports, debug e entrega. Por isso o objetivo é achar inconsistências cedo, antes que o design avance para etapas mais caras do fluxo.

A verificação tenta responder uma pergunta específica: o comportamento observado bate com a intenção descrita pela especificação? Essa pergunta é mais forte do que “o simulador terminou”. O simulador é apenas o motor. A verificação é o método.

Erro de especificação

A intenção está incompleta, ambígua ou errada. O RTL pode implementar fielmente algo que não resolve o problema real.

Erro de implementação

A especificação está correta, mas o RTL não implementa a função, temporização ou estado esperado.

Erro de verificação

O testbench não observa o bug, não checa a resposta, ou passa por falta de cobertura dos casos relevantes.

Causas comuns de bugs de design

Os slides destacam que bugs não vêm de uma única fonte. Eles podem aparecer por requisitos mal definidos, comunicação ruim entre equipes, interpretação errada de protocolo, corner cases esquecidos, problemas de clock, reset incompleto, estado inválido, erro de interface ou até erro no próprio testbench.

Causa Exemplo em RTL Como a verificação ajuda
Especificação ambígua Não está claro o que acontece se duas instruções disputam o mesmo recurso. Transforma a dúvida em caso de teste e pergunta explícita.
Corner case omitido Overflow, underflow, reset no meio da operação, instrução inválida. Cria testes direcionados e cobertura funcional.
Erro de temporização funcional Sinal válido um ciclo antes/depois do esperado. Monitores e assertions detectam protocolo de ciclos.
Erro de integração Módulo isolado funciona, mas falha conectado ao top. Ambientes hierárquicos e testes de sistema revelam incompatibilidades.

O que é verificação?

Verificação é comparar o design contra uma referência de comportamento. Essa referência pode ser uma especificação humana, um modelo de alto nível, um scoreboard, uma função de referência em C/SystemVerilog, uma assertion ou uma propriedade formal. O ponto é: a resposta do DUT precisa ser julgada por algo externo ao DUT.

1. Estímulo Aplica entradas, transações ou cenários ao DUT.
2. Observação Monitora sinais, protocolos, saídas e eventos relevantes.
3. Checagem Compara comportamento observado com o esperado.

Esse trio é a semente do ambiente reutilizável que você comentou antes: driver para aplicar estímulos, monitor para observar, scoreboard para comparar e coverage para medir se os cenários importantes foram exercitados.

Verificação é diferente de simulação

Simulação é uma técnica. Verificação é uma atividade de engenharia. Você pode simular muito e verificar pouco, se os testes não têm intenção clara ou se ninguém checa as respostas. Também pode simular pouco e verificar bem um caso específico, se o teste for dirigido e tiver checker robusto.

Simulação

Executa o modelo RTL ao longo do tempo, resolve eventos, calcula sinais e gera logs/waveforms. Ela mostra o que aconteceu.

Verificação

Decide se o que aconteceu está correto, com base em especificação, checadores, assertions, scoreboard e cobertura.

Frase para memorizar: waveform ajuda a investigar bug; não deve ser o principal mecanismo de dizer que o design está correto. O ideal é que o testbench acuse erro automaticamente.

Verification versus testing

No vocabulário do curso, verificação é feita antes da fabricação ou implantação final, normalmente no modelo RTL, gate-level ou formal. Testing, no sentido de teste de manufatura, acontece no hardware físico para detectar defeitos de fabricação. São atividades relacionadas, mas não iguais.

Termo Quando acontece O que procura
Verification Durante o projeto, antes da fabricação. Erros de design, especificação, RTL, integração e comportamento.
Testing Depois que o chip/dispositivo físico existe. Defeitos de manufatura, falhas físicas e problemas de produção.

Simuladores são modelos, não realidade

O simulador executa uma descrição do design. Isso tem duas consequências. Primeiro, ele só conhece o que está no modelo: se o RTL ou testbench não descreve uma condição, o simulador não inventa intenção. Segundo, ele trabalha com abstrações, não com o hardware real completo. Mesmo gate-level simulation com SDF ainda é uma aproximação.

Event-based

Simula mudanças de sinais e eventos no tempo. É detalhado e flexível, mas pode ser lento em designs grandes.

Cycle-based

Foca em bordas/ciclos relevantes. Pode ser mais rápido, mas menos adequado para detalhes finos de eventos.

Co-simulation

Integra modelos em linguagens/domínios diferentes, como HDL com C/C++ ou partes digitais com analógicas.

Por que o simulador não sabe sua intenção?

Se um protocolo diz que valid precisa permanecer alto até ready, o simulador não acusa erro sozinho quando isso é violado. Para ele, são só sinais mudando. Quem transforma isso em erro é o testbench, uma assertion ou um monitor que conhece a regra do protocolo.

Por que verificação exaustiva é impossível?

Os slides usam exemplos como unidade de divisão floating-point de 64 bits e memória de 256 bits para mostrar a explosão combinatória. Mesmo que cada simulação fosse rápida, testar todas as combinações de entradas, estados e sequências seria impraticável. Além disso, muitos bugs dependem da ordem temporal dos eventos, não apenas de uma combinação estática de bits.

espaço de estados = entradas × estados internos × sequências temporais × interações de ambiente

Por isso, verificação real usa estratégia: testes dirigidos para casos conhecidos, randomização controlada para explorar o espaço, coverage para medir progresso, assertions para protocolos e regressão para repetir cenários importantes.

Coverage Driven Verification

Coverage-driven verification é uma forma de não depender apenas de sensação. Você define o que precisa ser exercitado, roda testes, mede cobertura, encontra buracos e cria novos testes para fechar esses buracos. O ciclo continua até o plano estar suficientemente coberto e os checkers não acusarem erro.

Plano Lista funções, cenários, protocolos, corner cases e métricas.
Execução Roda testes dirigidos, randomizados e regressões.
Medição Analisa code coverage e functional coverage para decidir próximos testes.
Tipo de cobertura O que mede Armadilha
Code coverage Linhas, branches, expressões, toggles ou FSMs exercitados. Pode chegar a 100% sem provar que o comportamento esperado foi checado.
Functional coverage Cenários funcionais definidos no plano de verificação. Depende da qualidade do plano; se o plano esquece algo, a cobertura também esquece.
Cross coverage Combinações entre variáveis importantes. Pode explodir em número de bins se for mal planejada.

Falso positivo e falso negativo

Dois conceitos aparecem como perigo prático. Um falso positivo acontece quando o testbench diz que passou, mas o design ainda tem bug. Um falso negativo acontece quando o testbench acusa erro, mas o problema está no testbench, na expectativa ou na especificação usada como referência.

Falso positivo

Mais perigoso para entrega: você acredita que o design está correto. Normalmente nasce de checker fraco, pouca cobertura ou teste que só olha waveform manualmente.

Falso negativo

Gasta tempo de debug: o RTL pode estar correto, mas a referência está errada. Normalmente revela bug no testbench, scoreboard ou interpretação da especificação.

Como isso vira ambiente de verificação?

A ponte para os próximos blocos é natural. Uma verificação manual simples pode começar com testbench dirigido. Depois você separa responsabilidades para reaproveitar:

Driver

Transforma transações ou comandos em sinais no DUT.

Monitor

Observa sinais do DUT e reconstrói eventos/transações.

Scoreboard

Compara resultado observado com modelo esperado.

Coverage

Mede quais cenários foram exercitados e quais ainda faltam.

Para o laboratório do professor: o Makefile deve rodar simulação, mas o valor real vem do testbench acusar erro automaticamente e gerar artefatos de debug, como logs e FSDB. A próxima evolução é ter testes nomeados, regressão e cobertura, não só um comando que abre waveform.

Roteiro prático para nosso CPU

  1. Escrever quais comportamentos do CPU precisam ser verificados: reset, instruções, flags, memória, desvios e casos inválidos.
  2. Separar testes dirigidos por funcionalidade, em vez de um testbench único que faz tudo sem nome.
  3. Adicionar checagem automática: cada teste deve terminar com PASS/FAIL confiável.
  4. Gerar waveform só quando necessário para debug, não como critério principal de aprovação.
  5. Depois, adicionar coverage funcional para saber se todos os cenários importantes foram exercitados.

Revisão para fixar

1. Simulação e verificação são a mesma coisa? Não. Simulação executa o modelo; verificação decide se o comportamento observado está correto.
2. Code coverage 100% prova ausência de bugs? Não. Ela mostra que o código foi exercitado, não que todas as intenções funcionais foram checadas.
3. Por que functional coverage depende do plano? Porque ela só mede os cenários que alguém decidiu modelar. Plano incompleto gera cobertura incompleta.
4. O que é falso positivo em verificação? É quando o teste passa, mas o design ainda contém bug relevante.
5. Por que waveform não deve ser o verificador principal? Porque análise visual é manual, incompleta e difícil de repetir em regressão.
