Bases de RTL Sintetizável: `logic`, `always_comb`, `always_ff` e FSM

Esta aula transforma os primeiros self-checks do curso em uma leitura prática: como escrever SystemVerilog que o simulador aceita, que a síntese entende e que não cria hardware diferente do que você imaginou.

Objetivo desta aula

SystemVerilog não serve apenas para deixar o código "mais moderno". O ponto central é declarar intenção: combinacional com always_comb, sequencial com always_ff, exclusividade com unique, prioridade com priority, estado simbólico com enum e sinais procedurais com logic.

Essa intenção ajuda ferramentas e humanos. Mas ela também cria responsabilidade: se você promete que um case é unique quando não é, ou escreve always_comb deixando saída sem atribuição, a ferramenta pode avisar, otimizar de forma inesperada ou revelar diferença entre simulação e síntese.

Para o nosso projeto: este bloco vira checklist de revisão do CPU antes de automatizar síntese. Queremos módulos com portas claras, combinacional sem latch involuntário, registradores com reset consistente e FSMs legíveis.

Mapa mental da aula

1. Tipos e portas Entender `logic`, signed/unsigned, ANSI style e por que `output` sem tipo pode virar net.
2. Combinacional Usar `always_comb`, atribuir todos os caminhos e evitar latch involuntário.
3. Decisão Usar `priority` e `unique` como contratos, não como decoração.
4. Sequencial Usar `always_ff`, eventos corretos, reset claro e atribuição não bloqueante.
5. Estados Usar `enum` para FSMs mais legíveis e menos frágeis.

`logic` é tipo de dado, não promessa de registrador

O curso começa cutucando uma confusão comum. Em Verilog antigo, a divisão mental era muitas vezes wire para conexões contínuas e reg para valores atribuídos dentro de always. O nome reg sempre foi traiçoeiro, porque nem todo reg sintetiza registrador. Se você usa reg em um bloco combinacional completo, ele sintetiza lógica combinacional, não flip-flop.

SystemVerilog introduz logic como um tipo mais conveniente para sinais de RTL. Ele pode ser usado em muitos lugares onde antes se usava reg. Mas logic também não significa "registrador". Quem decide se o hardware será fio, porta combinacional, latch ou flip-flop é o contexto de atribuição.

Forma de código Intenção típica Hardware provável
assign y = a & b; Conexão contínua combinacional. Portas/lógica combinacional.
always_comb y = a & b; Bloco procedural combinacional. Portas/lógica combinacional.
always_comb com caminho sem atribuição Erro de intenção: faltou cobrir caso. Latch, warning ou comportamento suspeito.
always_ff @(posedge clk) Atualização em borda de clock. Flip-flop.
Self-check: `logic unsigned` recebendo `-1`

O slide pergunta se este código é legal:

logic [7:0] unsigned data;
data = -1;

Sim, é legal. O valor -1 é representado em complemento de 2 como todos os bits em 1. Em 8 bits, isso é 8'b1111_1111. Como a variável é unsigned, se você interpretar esse padrão como número sem sinal, ele vale 255.

A lição é importante: signed e unsigned mudam interpretação aritmética e comparação. O hardware continua guardando bits. Prova adora esse tipo de pergunta porque a resposta não é "não pode guardar negativo"; a resposta é "guarda o padrão de bits, depois interpreta conforme o tipo".

Portas: escolha ANSI style e seja explícito

Outro self-check do curso mostra uma declaração misturada: 

module my_module (clk, reset, logic din, output logic d1, d2);

Isso está errado porque mistura dois estilos de declaração de portas. Se você começa listando apenas nomes (clk, reset), está usando o estilo non-ANSI. Nesse estilo, direção e tipo devem aparecer depois, no corpo do módulo. Se você quer colocar direção e tipo dentro da lista de portas, use ANSI style desde o começo.

Non-ANSI style

module my_module (clk, reset, din, d1, d2);
  input  clk, reset, din;
  output logic d1, d2;
endmodule

Funciona, mas separa nome, direção e tipo. Em projetos maiores, fica mais fácil errar.

ANSI style

module my_module (
  input  logic clk,
  input  logic reset,
  input  logic din,
  output logic d1,
  output logic d2
);
endmodule

É mais claro, revisável e combina melhor com RTL moderno.

Regra para nosso CPU: ao adaptar módulos, prefira ANSI style. Cada porta deve dizer direção, tipo e largura no mesmo lugar. Isso ajuda revisão, síntese, lint e conexão por nome.

`output` sem tipo pode virar net

O curso traz outra pegadinha prática. Se você declara uma saída apenas como output d1, sem tipo, a linguagem pode tratar essa porta como uma net, similar a wire. Uma net é dirigida por atribuição contínua ou por instância, não por atribuição procedural dentro de always. 

// Problemático: d1 e d2 são outputs sem tipo explícito
module my_module (
  input clk,
  input reset,
  input din,
  output d1,
  output d2
);
  always @(posedge clk) begin
    d1 <= din;
    d2 <= d1;
  end
endmodule

Se d1 e d2 forem nets, o bloco always não deve dirigi-los. A forma correta para um sinal atualizado em bloco procedural é declarar como variável, por exemplo output logic. 

// Melhor: portas explícitas e bloco sequencial explícito
module my_module (
  input  logic clk,
  input  logic rst_n,
  input  logic din,
  output logic d1,
  output logic d2
);
  always_ff @(posedge clk or negedge rst_n) begin
    if (!rst_n) begin
      d1 <= 1'b0;
      d2 <= 1'b0;
    end else begin
      d1 <= din;
      d2 <= d1;
    end
  end
endmodule
Por que isso importa tanto?

Porque a declaração de porta é a fronteira do módulo. Se ela está vaga, cada ferramenta precisa aplicar regras padrão. Regras padrão são perigosas para iniciante porque o código parece simples, mas o objeto criado não é o que você imaginou. Em RTL didático e em projeto de curso, clareza vale mais que economia de caracteres.

`always_comb`: combinacional com contrato

always_comb é a forma moderna de declarar que um bloco é combinacional. Ele substitui o uso clássico de always @(*) em muitos casos e comunica intenção para ferramentas: este bloco não deveria ter memória. Portanto, toda saída atribuída dentro dele precisa receber valor em todos os caminhos possíveis. 

// Perigoso: se sel não ativa nenhum if, d mantém valor anterior
always_comb begin
  if (sel[0])
    d = a;
  else if (sel[1])
    d = b;
  else if (sel[2])
    d = c;
end

O problema é simples e profundo: lógica combinacional não "lembra" valor anterior. Se o código deixa um caminho sem atribuição, a síntese precisa preservar o valor antigo. Preservar valor antigo exige memória. Essa memória normalmente vira latch. 

// Melhor: default primeiro, depois sobrescreve nos casos relevantes
always_comb begin
  d = 1'b0;

  if (sel[0])
    d = a;
  else if (sel[1])
    d = b;
  else if (sel[2])
    d = c;
end

Default no topo

Garante valor mesmo quando nenhum caso específico acontece.

Todos os caminhos

Cada saída do bloco precisa ser atribuída independentemente do caminho lógico.

Latch involuntário

Quase sempre indica que a intenção combinacional foi escrita de forma incompleta.

Como revisar um `always_comb` no nosso projeto?
  1. Liste todas as saídas atribuídas no bloco.
  2. Veja se cada uma recebe default antes de qualquer if ou case.
  3. Se não houver default, confira se todo if tem else completo.
  4. Em case, confira se existe default ou se há justificativa real para unique/priority.
  5. Se aparecer latch warning, trate como bug até provar o contrário.

`full` e `parallel`: duas perguntas diferentes

O curso introduz os conceitos de full e parallel. Eles parecem abstratos, mas são duas perguntas bem concretas sobre decisões em RTL.

Conceito Pergunta Risco quando falso
Full Todos os valores/caminhos possíveis foram cobertos? Caminho sem atribuição, latch, default indesejado ou warning.
Parallel Os ramos são mutuamente exclusivos? Lógica de prioridade inesperada ou otimização incorreta se você prometeu exclusividade.

Um case pode ser full sem ser parallel, parallel sem ser full, os dois, ou nenhum dos dois. O erro de iniciante é achar que uma palavra-chave resolve os dois problemas automaticamente.

`priority` e `unique` são promessas

priority e unique não são enfeites de estilo. Eles dizem ao simulador e à síntese como você entende a decisão.

`priority`

Indica que existe ordem de preferência. Se mais de uma condição for verdadeira, a primeira ganha. É natural para decodificadores com prioridade, tratamento de interrupções ou regras do tipo "caso especial antes do caso geral". 

always_comb begin
  grant = 3'b000;

  priority if (req[0])
    grant = 3'b001;
  else if (req[1])
    grant = 3'b010;
  else if (req[2])
    grant = 3'b100;
end

`unique`

Indica que no máximo uma alternativa deveria casar. É natural para seleção mutuamente exclusiva, estados de FSM e decodificação em que os códigos não deveriam sobrepor. 

always_comb begin
  next_state = state;

  unique case (state)
    IDLE: if (start) next_state = BUSY;
    BUSY: if (done)  next_state = DONE;
    DONE:           next_state = IDLE;
  endcase
end
Cuidado: se você usar unique em condições que podem sobrepor, você está mentindo para a ferramenta. Isso pode gerar warning em simulação e pode permitir otimizações que não refletem a prioridade que você esperava.
Por que não resolver tudo colocando `default: 'x`?

O slide chama atenção para um padrão perigoso: usar 'x como default para "ajudar" a ferramenta. Em simulação, X pode revelar estado inválido. Em síntese, X pode virar don't-care, permitindo otimização. Isso é útil em casos controlados, mas perigoso como hábito.

Em código de aluno, CPU didático e laboratório, a melhor regra é: use default real e seguro, trate estados inválidos de forma explícita, e deixe 'x para situações em que você sabe exatamente o efeito na simulação e na síntese.

`always_ff`: sequencial com evento obrigatório

always_ff declara intenção de flip-flop. Isso significa que o bloco precisa ter controle de evento adequado, normalmente borda de clock e, quando necessário, borda de reset assíncrono. 

// Registrador simples com reset assíncrono ativo baixo
always_ff @(posedge clk or negedge rst_n) begin
  if (!rst_n)
    q <= '0;
  else
    q <= d;
end

Um self-check do curso mostra que always_ff sem event control está errado. Faz sentido: se você diz "isso é flip-flop", precisa dizer em qual borda ele atualiza. 

// Errado para always_ff: não há evento de clock
always_ff begin
  q <= d;
end
Regra de ouro: bloco sequencial usa atribuição não bloqueante, isto é, <=. Bloco combinacional normalmente usa atribuição bloqueante, isto é, =.
Por que `<=` em registrador?

Com atribuição não bloqueante, todos os registradores do bloco amostram seus valores antigos e atualizam juntos no fim do passo de simulação. Isso imita melhor o comportamento de flip-flops reais em uma borda de clock. 

always_ff @(posedge clk) begin
  a <= b;
  b <= a;
end

O exemplo acima troca a e b em uma borda. Se você usar = sem entender o escalonamento da simulação, pode criar comportamento de simulação diferente do hardware pretendido.

Reset assíncrono e reset síncrono

O curso entra em exemplos de registrador porque reset é um dos lugares onde RTL bonito pode virar hardware inesperado. A diferença central é esta:

Reset assíncrono

O reset aparece na lista de sensibilidade por borda. O registrador pode ser resetado independentemente da borda de clock. 

always_ff @(posedge clk or negedge rst_n) begin
  if (!rst_n)
    q <= '0;
  else
    q <= d;
end

Reset síncrono

A lista de sensibilidade tem apenas o clock. O reset é testado dentro do bloco e só tem efeito na borda de clock. 

always_ff @(posedge clk) begin
  if (!rst_n)
    q <= '0;
  else
    q <= d;
end

O detalhe que aparece nos slides é que certas formas ambíguas de set/reset podem ser interpretadas como assíncronas pela ferramenta. Quando o código não deixa claro qual sinal é clock, qual é reset e qual é set, a síntese precisa inferir intenção. Inferência demais é terreno fértil para surpresa.

No CPU: escolha uma política e mantenha consistência. Se o projeto usa reset ativo baixo assíncrono, use o mesmo padrão em todos os registradores principais. Se usa reset síncrono, deixe isso claro nos templates e nos testes.

Enums tornam FSMs estudáveis

A parte A fecha em variáveis enumeradas. Em Verilog puro, muitos alunos escrevem estados como parâmetros: 

localparam IDLE = 2'b00;
localparam BUSY = 2'b01;
localparam DONE = 2'b10;

Isso funciona, mas SystemVerilog permite declarar um tipo de estado mais expressivo: 

typedef enum logic [1:0] {
  IDLE,
  BUSY,
  DONE,
  ERROR
} state_t;

state_t state, next_state;

Agora state e next_state carregam significado no tipo. O waveform pode mostrar nomes, o código fica mais legível e a revisão fica mais direta. Para uma controladora do CPU, isso é muito melhor do que comparar números mágicos espalhados. 

always_comb begin
  next_state = state;

  unique case (state)
    IDLE:  if (start) next_state = BUSY;
    BUSY:  if (done)  next_state = DONE;
    DONE:            next_state = IDLE;
    ERROR:           next_state = IDLE;
    default:         next_state = ERROR;
  endcase
end

always_ff @(posedge clk or negedge rst_n) begin
  if (!rst_n)
    state <= IDLE;
  else
    state <= next_state;
end
O padrão de duas partes para FSM

Uma FSM didática costuma separar registrador de estado e lógica de próximo estado. Isso deixa claro o que é sequencial e o que é combinacional.

Registrador de estado always_ff, atualiza state na borda de clock.
Próximo estado always_comb, calcula next_state a partir do estado atual e das entradas.
Saídas Podem ser combinacionais ou registradas, conforme a arquitetura escolhida.

Checklist para aplicar no nosso RTL

Esta aula vira um checklist de revisão antes de rodar síntese ou encaixar o design no ambiente do professor.

Portas Usar ANSI style, direção explícita, largura explícita e output logic quando a saída é dirigida por bloco procedural.
Combinacional Preferir always_comb, atribuir defaults e remover latch warnings.
Decisões Usar priority apenas quando há prioridade real; usar unique apenas quando há exclusividade real.
Registradores Usar always_ff, event control correto e <= para atualização sequencial.
Reset Separar claramente reset síncrono de assíncrono e manter padrão no projeto.
FSM Usar enum, state/next_state e tratamento explícito de estado inválido.

Revisão para prova e laboratório

1. `logic [7:0] unsigned data; data = -1;` é ilegal? Não. O padrão de bits armazenado é 8'b1111_1111. Interpretado como unsigned, vale 255.
2. `logic` significa que a síntese criará registrador? Não. O contexto decide. always_ff com borda cria registrador; always_comb completo cria lógica combinacional.
3. Por que `output d1` pode ser problema dentro de `always`? Porque output sem tipo pode criar net. Net não deve ser dirigida por atribuição procedural. Use output logic quando o sinal é atribuído dentro de bloco procedural.
4. Qual é o sintoma clássico de combinacional incompleto? Latch involuntário. Em geral, faltou default ou faltou cobrir um caminho de if/case.
5. `priority` e `unique` significam a mesma coisa? Não. priority expressa ordem de preferência. unique expressa exclusividade entre alternativas.
6. Reset assíncrono aparece onde? Na lista de eventos do always_ff, por exemplo @(posedge clk or negedge rst_n).
