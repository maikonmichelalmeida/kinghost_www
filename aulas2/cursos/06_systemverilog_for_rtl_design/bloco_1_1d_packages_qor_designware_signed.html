Raiz/06 SystemVerilog for RTL Design/QoR e fechamento

Packages, QoR, DesignWare e signed arithmetic

Este bloco fecha o curso 06. A mensagem final é direta: RTL não é texto que apenas “compila”; RTL é uma descrição de hardware, e pequenas escolhas de SystemVerilog podem mudar área, timing, nomes de netlist, simulação gate-level e capacidade de integração no fluxo Synopsys.

Objetivo desta aula

Até aqui, o curso ensinou a escrever sinais, blocos combinacionais, registradores, FSMs, interfaces e wrappers. O bloco final mostra o que acontece quando esse RTL chega mais perto da síntese real: parâmetros precisam ser elaborados, packages precisam ser analisados na ordem certa, DesignWare precisa estar linkado, e o estilo de codificação influencia QoR.

Esta aula também amarra o que importa para o laboratório do professor: se o ambiente dele automatiza análise, simulação e síntese com Makefiles/scripts, então nosso código precisa ter fronteiras previsíveis, dependências claras, pacotes bem organizados e datapaths escritos de forma que a ferramenta entenda a intenção.

Visão final do curso 06

O curso 06 fecha um arco completo:

Bloco 2 FSMs, `unique0`, `X`, wildcards, arrays e structs.

Amarração: o SystemVerilog bom para RTL é aquele que deixa explícito o hardware pretendido e reduz o trabalho de adivinhação da ferramenta. Quanto menos a ferramenta precisa inferir intenção, mais previsível fica o fluxo de simulação, síntese e debug.

Wrappers e parâmetros: fechando a história

A parte D retoma wrappers porque eles são uma resposta prática para uma diferença inevitável: o RTL que você escreve pode ser bonito e abstrato; a netlist sintetizada precisa ter nomes, portas e instâncias concretas. 

module fifo #(WIDTH = 8, BUF_SIZE = 16)
  (input logic clk,
   input logic reset_n,
   fifo_io.fifo fifo_if);
endmodule

Quando você elabora uma versão com parâmetros não default, por exemplo WIDTH=4 e BUF_SIZE=8, a ferramenta pode criar um design especializado. O nome final pode deixar de ser apenas fifo e virar algo como: 

fifo_WIDTH4_BUF_SIZE8_I_fifo_if_fifo_io__4

Além disso, a porta de interface pode ser achatada em portas reais, como fifo_if.rd_n, fifo_if.wr_n, fifo_if.empty e assim por diante. O wrapper entra para dar contexto ao bloco durante a elaboração e para permitir que o script descubra o nome real do design. 

analyze -format sverilog { fifo_io.sv fifo.sv wrapper_fifo.sv }
elaborate wrapper_fifo -parameter "WIDTH=4, BUF_SIZE=8"
current_design [get_designs fifo*]
Por que o wildcard aparece? porque o nome do módulo pode ter sido modificado pela especialização de parâmetros. Usar current_design fifo pode falhar; current_design [get_designs fifo*] encontra a versão elaborada.
Como isso conversa com gate-level simulation?

Depois da síntese, o testbench ou wrapper antigo pode tentar instanciar fifo, enquanto a netlist contém fifo_WIDTH4_BUF_SIZE8.... Também pode tentar conectar uma interface, enquanto a netlist tem portas achatadas. Por isso o fluxo precisa gerar ou atualizar um wrapper de simulação compatível com a netlist real.

Integrando netlist de bloco no top-level

A aula mostra um cenário comum em fluxo bottom-up: você sintetiza um bloco isolado e depois quer integrá-lo no design maior. A sequência conceitual é:

  1. Analisar o RTL completo do top-level.
  2. Elaborar o top-level.
  3. Remover o RTL do bloco que será substituído pela netlist sintetizada.
  4. Ler a netlist gate-level daquele bloco.
  5. Linkar o design para resolver referências.
analyze -format sverilog { fifo_io.sv fifo.sv top.sv }
elaborate top
remove_design fifo
read_verilog fifo_mapped.v
link

A parte importante para nós não é decorar comandos isolados, mas entender o motivo: se você mantém o RTL original e também lê a netlist do mesmo bloco, a ferramenta pode ter duas definições concorrentes. O fluxo precisa dizer com clareza: neste ponto, use a versão sintetizada.

Packages: biblioteca de definições, não hardware global

Packages servem para compartilhar declarações entre módulos, interfaces e outros elementos SystemVerilog. Eles são excelentes para constantes, tipos, enums, structs, funções utilitárias e parâmetros locais de configuração. 

package cpu_pkg;
  localparam int XLEN = 32;

  typedef enum logic [2:0] {
    ALU_ADD,
    ALU_SUB,
    ALU_AND,
    ALU_OR
  } alu_op_t;

  typedef struct packed {
    logic    valid;
    alu_op_t alu_op;
    logic    reg_write;
    logic    mem_read;
    logic    mem_write;
  } control_t;
endpackage

Isso evita repetir a mesma definição em vários arquivos. Se todos os módulos usam o mesmo alu_op_t, o decode, a ALU e o testbench ficam mais alinhados. Mas o package não é um módulo de hardware e não deve virar um depósito de estado global.

Bom uso de package Evitar em RTL sintetizável
localparam, typedefs, enums, structs. Declarar wire, logic variável ou estado físico no escopo do package.
Funções utilitárias automatic. always, initial ou assign dentro do package.
Definições autocontidas e fáceis de importar. Dependências implícitas difíceis de resolver no script.
Pegadinha de prova: parâmetros em package devem ser localparam. Eles não são parâmetros configuráveis por instância como em um módulo.

Importar package no escopo correto

O curso também diferencia acesso explícito e importação. Você pode acessar um item com qualificação: 

cpu_pkg::alu_op_t op;

Ou importar itens para o escopo: 

import cpu_pkg::*;

alu_op_t op;

O estilo explícito é mais verboso, mas deixa claro de onde veio o nome. O import global é mais conveniente, mas aumenta risco de colisão. Em projeto pequeno, ambos aparecem; em projeto maior, vale ser mais disciplinado.

Fluxo de Design Compiler para packages

Se o RTL depende de package, o package precisa ser analisado antes e precisa estar no caminho/biblioteca que a ferramenta consegue encontrar. 

define_design_lib pkg_lib -path ./pkg_lib
lappend search_path { ./package ./pkg_lib }

analyze -format sverilog -library pkg_lib { pkg.sv }
analyze -format sverilog { encryptor.sv }
elaborate encryptor
link
compile

Funções em packages precisam ser `automatic`

O slide de regras diz que tasks e functions em packages devem ser automatic. A razão didática é que uma função em package deve se comportar como lógica combinacional reutilizável, sem estado compartilhado implícito entre chamadas. 

package math_pkg;
  function automatic logic [3:0] sat_add4 (
    input logic [3:0] a,
    input logic [3:0] b
  );
    logic [4:0] tmp;
    tmp = a + b;
    sat_add4 = tmp[4] ? 4'hF : tmp[3:0];
  endfunction
endpackage

Esse tipo de função pode deixar o RTL mais limpo quando a operação é pequena e aparece em vários lugares. Mas não transforme package em lugar para processos, registradores ou sinais globais. O hardware deve estar em módulos e interfaces, não perdido no escopo do package.

QoR: estilo de código vira hardware diferente

QoR significa Quality of Results. No contexto de síntese, estamos olhando principalmente para área, timing, contagem de células, uso de memória e tempo de execução da ferramenta. A aula usa uma função simples para mostrar algo profundo: rotacionar um vetor à esquerda pode ser descrito de várias formas, mas essas formas não geram necessariamente o mesmo hardware.

Shift left

Desloca bits e preenche posições novas com zero.

out = din << k;

Rotate left

Desloca bits e reinjeta os bits que saíram no outro lado.

out = (din << k) | (din >> (WIDTH-k));

O curso compara estilos como assign, for loop, while, case, shift e componente DesignWare. A conclusão é que estilo não é estética: dois códigos equivalentes em simulação podem induzir estruturas diferentes no sintetizador.

Clareza funcional

A ferramenta entende melhor quando o código expressa diretamente a operação pretendida.

QoR medido

Área e timing precisam ser lidos em reports, não presumidos só pela aparência do código.

Loop não é software

Loop em RTL geralmente descreve hardware replicado ou lógica combinacional expandida.

Loops em RTL não rodam no tempo como software

Quando um aluno vê for, a tendência é imaginar um processador executando iterações. Em RTL sintetizável, dentro de um always_comb, um loop geralmente é desenrolado pela síntese para criar lógica combinacional. Ele é uma forma compacta de descrever hardware repetido. 

always_comb begin
  out = '0;

  for (int i = 0; i < WIDTH; i++) begin
    out[(i + k) % WIDTH] = din[i];
  end
end

Esse código pode ser funcionalmente correto para rotate, mas pode gerar lógica menos óbvia do que uma formulação com shifts e OR. A ferramenta pode otimizar, mas a forma inicial de descrição influencia a inferência.

Regra mental: em RTL, pergunte sempre “que hardware isso descreve?” antes de perguntar “esse algoritmo funciona?”.

DesignWare: usar biblioteca quando faz sentido

DesignWare é uma biblioteca de componentes parametrizáveis da Synopsys. Em vez de confiar apenas na inferência, o designer pode instanciar um componente conhecido, como um barrel shifter. 

// Exemplo conceitual de instância DesignWare
DW01_bsh #(
  .A_width(WIDTH),
  .SH_width($clog2(WIDTH))
) u_shifter (
  .A(din),
  .SH(k),
  .B(out)
);

A vantagem é usar uma implementação parametrizável reconhecida pelo fluxo. O cuidado é que a simulação e a síntese precisam enxergar as bibliotecas DesignWare. 

// Simulação: apontar modelos DesignWare
-y ${SYNOPSYS}/dw/sim_ver +libext+.v+.sv +incdir+${SYNOPSYS}/dw/sim_ver+

// Síntese: linkar biblioteca DesignWare
set_app_var link_library "* $target_library dw_foundation.sldb"

No exemplo do curso, a inferência por estilos bem escritos conseguiu chegar a resultado equivalente ao componente DesignWare. Isso também é uma lição: DesignWare é ferramenta poderosa, mas bom RTL inferível continua sendo muito valioso.

Timing e área podem melhorar juntos?

Uma ideia comum é imaginar que melhorar timing sempre aumenta área, e reduzir área sempre piora timing. Muitas vezes existe tradeoff, sim. Mas o slide pergunta se bom timing e boa área podem existir ao mesmo tempo. A resposta prática é: podem, quando o estilo de RTL permite que a ferramenta encontre uma estrutura melhor.

Um código ruim pode ser simultaneamente maior e mais lento. Um código mais direto pode permitir uma implementação menor e com caminho crítico melhor. Por isso, quando olhamos reports, não devemos apenas apertar constraints; às vezes o problema está no modo como o RTL descreveu o datapath.

Para o CPU: se a síntese mostrar área/timing ruins em ALU, shifter, muxes ou controle, não pense só em constraint. Revise o estilo RTL e veja se a operação está escrita de forma clara para o sintetizador.

Signed arithmetic: deixe a linguagem expressar a intenção

O curso fecha com aritmética signed em datapath, um tema pequeno que pode gerar diferença enorme de hardware. O erro mostrado é fazer extensão manual de sinal e, sem querer, transformar uma multiplicação signed 8x8 em uma multiplicação unsigned 16x16. 

// Pior QoR: extensão manual pode induzir multiplicação maior
logic [15:0] a_ext;
logic [15:0] b_ext;
logic [15:0] z;

assign a_ext = {{8{a[7]}}, a};
assign b_ext = {{8{b[7]}}, b};
assign z     = a_ext * b_ext; // pode ser tratado como unsigned 16x16

A forma melhor é usar os recursos signed de SystemVerilog, seja com casts, seja declarando as portas como signed. 

// Melhor: cast signed
logic signed [15:0] z_sgn;

assign z_sgn = signed'(a) * signed'(b);
assign z     = unsigned'(z_sgn);
// Melhor ainda quando a interface do bloco permite
module mul_signed (
  input  logic signed [7:0]  a,
  input  logic signed [7:0]  b,
  output logic signed [15:0] z
);
  assign z = a * b;
endmodule
Regra final de datapath: se a operação é signed, diga isso ao tipo. Não force a ferramenta a deduzir intenção a partir de concatenações manuais.

Fechamento didático do curso

Agora o curso 06 fica amarrado em uma regra só: SystemVerilog permite escrever intenção. Quando você declara bem a intenção, ganha legibilidade, warnings melhores, síntese mais previsível e integração mais limpa. Quando você deixa intenção implícita, a ferramenta precisa inferir, e a inferência pode aparecer como latch, mux estranho, nome de netlist inesperado, área maior, timing pior ou simulação gate-level difícil.

Dentro do módulo Use `logic`, `always_comb`, `always_ff`, enums, defaults e resets consistentes.
Na decisão Use `unique`, `priority`, `unique0`, `==?` e `case inside` apenas quando a intenção for real.
Na conexão Use interfaces/modports quando vários sinais formam um protocolo; use wrappers para fluxo controlado.
Na organização Use packages para tipos e constantes, sem criar estado global.
No datapath Escreva operações de forma que a ferramenta reconheça shifters, multiplicadores e signed arithmetic.
No fluxo Leia reports de QoR e revise RTL antes de culpar apenas constraints.

Aplicação prática no nosso próximo passo

Para o nosso CPU/laboratório, este curso sugere uma ordem madura de evolução:

  1. Revisar o RTL atual procurando latches involuntários, resets inconsistentes e portas pouco explícitas.
  2. Padronizar FSMs com enum, largura explícita e defaults em `always_comb`.
  3. Separar constantes e tipos compartilhados em package somente quando isso reduzir duplicação real.
  4. Criar wrappers quando começarmos a sintetizar blocos de forma isolada ou parametrizada.
  5. Usar Makefiles para automatizar análise, elaboração, simulação, síntese e abertura de reports.
  6. Ler QoR e ajustar estilo RTL quando área/timing indicarem problema.
Transição natural: o próximo curso, Design Compiler NXT / RTL Synthesis, deve aprofundar exatamente a etapa seguinte: ler, elaborar, linkar, aplicar constraints, compilar e interpretar reports.

Revisão final

1. Por que usar wrapper em síntese bottom-up? Para instanciar interface, parâmetros e DUT em um contexto controlado, e para lidar com nomes/portas que mudam na síntese.
2. O que pode acontecer com um módulo parametrizado? A ferramenta pode gerar um nome especializado, como uma versão de `fifo` com `WIDTH` e `BUF_SIZE` embutidos.
3. Package pode declarar `logic state;` no escopo global para síntese? Não. Package deve guardar definições reutilizáveis, não estado físico global.
4. O que deve vir antes no script: package ou RTL que usa package? O package deve ser analisado primeiro e estar no search path/biblioteca correta.
5. O que é QoR? Quality of Results: métricas como área, timing, células, nets, memória e tempo de execução da síntese.
6. DesignWare substitui bom RTL? Não. Ela fornece componentes otimizados, mas RTL bem escrito pode inferir estruturas equivalentes em alguns casos.
7. Qual a regra para signed arithmetic? Declare ou caste como `signed` para expressar a intenção, evitando extensão manual que induz hardware maior.
