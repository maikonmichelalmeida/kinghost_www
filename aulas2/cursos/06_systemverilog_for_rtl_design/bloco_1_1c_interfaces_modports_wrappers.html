Interfaces, modports, parâmetros e wrappers

Este bloco fecha o tema de `union`/arrays/casts e entra em uma ideia central para projetos maiores: usar interfaces para organizar conectividade, modports para definir papéis, e wrappers para controlar a síntese e a simulação gate-level.

Objetivo desta aula

A parte C do curso mostra que SystemVerilog não é apenas uma linguagem para escrever lógica interna. Ela também oferece formas de organizar a comunicação entre módulos. Isso é muito útil quando o design começa a ter barramentos, sinais de controle, sinais de handshake e parâmetros espalhados.

Mas existe uma tensão importante: a interface é bonita no RTL, enquanto a netlist sintetizada precisa virar portas concretas. A ferramenta achata interfaces, renomeia sinais e pode até alterar o nome do módulo quando parâmetros não default são usados. Por isso entram os wrappers: eles preservam uma casca controlada para síntese bottom-up, integração e simulação gate-level.

Conexão com nosso laboratório: quando formos adaptar o CPU para um ambiente mais próximo do professor, wrappers e Makefiles vão andar juntos. O Makefile roda fluxo; o wrapper estabiliza a fronteira do bloco.

`union`: várias interpretações dos mesmos bits

A aula anterior abriu `union`; aqui o assunto fecha. A comparação com C é direta: uma union permite olhar para o mesmo armazenamento por visões diferentes. Em hardware, isso pode significar ler os mesmos 64 bits como um vetor bruto, como bytes ou como campos de cabeçalho. 

typedef union packed {
  tcp_t          tcp_h;
  udp_t          udp_h;
  logic [63:0]   bits;
  logic [7:0][7:0] bytes;
} header_u;

header_u h;

assign h.bits[31:16] = 16'h0005;

Se h.bits e h.bytes compartilham o mesmo armazenamento, alterar uma visão altera a outra. Isso pode ser excelente para reinterpretar pacote, instrução ou palavra de controle. Mas exige disciplina: quem lê o código precisa saber qual interpretação é válida em cada contexto.

Recurso Modelo mental Cuidado
struct packed Campos lado a lado, formando um vetor contínuo. Cada campo tem seu próprio pedaço do vetor.
union packed Várias visões sobre os mesmos bits. Todos os membros precisam ter tamanho compatível.
Regra do slide: em uma union packed, os membros devem ter o mesmo tamanho. Se os tamanhos não batem, a sobreposição dos bits fica ambígua para ferramenta e para humanos.

Packed e unpacked: a regra visual

A parte C volta aos arrays porque essa é uma pegadinha recorrente. A regra visual é simples:

Colchetes antes do nome

Dimensão packed. O sinal se comporta como palavra de bits contínua.

logic [7:0] value;

value = 8'b1010_1100;

Colchetes depois do nome

Dimensão unpacked. O sinal é coleção de elementos.

logic array [0:7];

array[0] = 1'b1;

O self-check do curso mostra que isto não é a mesma coisa: 

logic my_array [0:7];
my_array = 8'b1; // ilegal como atribuição escalar simples ao array inteiro

my_array é uma coleção de oito elementos de 1 bit. Já logic [7:0] my_value é uma palavra packed de 8 bits, então pode receber um valor único. 

logic [7:0] my_value;
my_value = 8'b0000_0001; // legal
Quando isso pega no CPU?

Em memórias e bancos de registradores. Por exemplo, logic [31:0] regfile [0:31] significa 32 elementos unpacked, cada um com 32 bits packed. Você pode atribuir um registrador inteiro: 

regfile[rd] <= write_data;

Mas não trate o banco inteiro como se fosse um vetor packed gigante sem conversão/estruturação explícita.

Cast explícito: dizer à ferramenta o que você quer

Quando você precisa converter entre uma coleção unpacked e uma palavra packed, o curso mostra a ideia de cast explícito. A sintaxe geral é: 

tipo_destino'(expressao)

O exemplo do material trabalha com um tipo packed de 8 bits e um tipo unpacked de 8 elementos: 

typedef logic [7:0] value_t;
typedef logic       array_t [8];

value_t value;
array_t array;

value = value_t'(array);

O detalhe conceitual é mais importante que decorar a sintaxe: se os dados têm organizações diferentes, não conte com conversão implícita mágica. Em RTL, conversão implícita obscura é convite para warning, mismatch ou código que passa na simulação e complica na síntese.

Regra prática: packed vector pode receber palavra de bits; unpacked array deve ser preenchido por elemento, por assignment pattern ou por cast explícito quando o fluxo suporta e a intenção é clara.

Interface: encapsular comunicação

O curso apresenta interface com uma analogia boa: uma struct agrupa dados relacionados; uma interface agrupa sinais de comunicação relacionados. Sem interface, um barramento simples começa a espalhar sinais por todos os módulos: req, gnt, addr, mode, start, rdy, dados, enables e assim por diante. 

// Sem interface: muitos sinais soltos na fronteira
module cpu_core (
  output logic        req,
  input  logic        gnt,
  output logic [7:0]  addr,
  output logic [1:0]  mode,
  output logic        start,
  input  logic        rdy
);
endmodule

Com interface, esse conjunto vira um objeto de comunicação. O módulo não recebe seis portas soltas; recebe uma interface que representa o barramento. 

interface simple_bus;
  logic        req;
  logic        gnt;
  logic [7:0]  addr;
  logic [1:0]  mode;
  logic        start;
  logic        rdy;
endinterface

module cpu_core (simple_bus bus);
  assign bus.req   = 1'b1;
  assign bus.addr  = 8'h20;
endmodule

Menos portas soltas

O top-level conecta uma interface em vez de repetir vários sinais manualmente.

Mais intenção

A interface dá nome ao protocolo ou conjunto de comunicação.

Mais responsabilidade

Na síntese, essa abstração será achatada em portas reais.

Modports: a interface vista por cada papel

Uma interface agrupa sinais, mas ainda falta definir quem dirige o quê. O modport resolve isso: ele define a visão da interface para cada tipo de módulo. O master pode dirigir req, addr e start; o slave pode dirigir gnt e rdy. 

interface simple_bus;
  logic        req;
  logic        gnt;
  logic [7:0]  addr;
  logic [1:0]  mode;
  logic        start;
  logic        rdy;

  modport master (
    output req, addr, mode, start,
    input  gnt, rdy
  );

  modport slave (
    input  req, addr, mode, start,
    output gnt, rdy
  );
endinterface

module cpu_core (simple_bus.master bus);
endmodule

module memory_block (simple_bus.slave bus);
endmodule

Essa separação é didaticamente poderosa: a mesma interface física tem os mesmos sinais, mas cada módulo tem um papel. Com modport, a ferramenta também consegue checar erros de direção mais cedo.

Papel Dirige Lê
Master/CPU req, addr, mode, start gnt, rdy
Slave/memória gnt, rdy req, addr, mode, start
Cuidado com `inout` dentro de interface

Um self-check do curso chama atenção para inout. Dentro de módulo e interface, um logic declarado internamente é variável por padrão. Para sinais bidirecionais reais, especialmente tri-state, você precisa pensar em net, drivers e atribuição contínua. Em muitos designs internos, evite bidirecionalidade e prefira sinais separados ou protocolo claro de direção.

O que a síntese faz com interfaces?

Esta é a parte mais importante para conectar o curso ao fluxo Synopsys. Interface é uma abstração de SystemVerilog. A netlist final precisa ter portas concretas. Portanto, a ferramenta de síntese achata a interface: os sinais internos viram portas individuais, muitas vezes com nomes derivados da instância da interface. 

interface fifo_io #(parameter WIDTH = 8);
  logic              rd_n;
  logic              wr_n;
  logic [WIDTH-1:0]  din;
  logic              empty;

  modport fifo (
    input  rd_n, wr_n, din,
    output empty
  );
endinterface

No RTL, o módulo pode ter uma porta de interface. Depois da síntese, a netlist pode ter portas achatadas como: 

fifo_if_rd_n
fifo_if_wr_n
fifo_if_din
fifo_if_empty

O nome exato depende do fluxo e das opções. A lição não é decorar o nome; é entender a consequência: o wrapper, o top-level e a simulação gate-level precisam estar alinhados com a forma que a síntese gerou.

Resumo importante: interface melhora o RTL, mas não sobrevive como "objeto de interface" na netlist tradicional. Ela vira portas.

Parâmetros não default complicam nomes

Quando o design usa parâmetros, a ferramenta precisa sintetizar uma versão especializada daquele módulo. Se você usa valores não default, como WIDTH=4 e BUF_SIZE=8, a ferramenta pode gerar um design com nome especializado. 

elaborate fifo -parameter "WIDTH=4, BUF_SIZE=8"

O curso mostra que o nome final pode virar algo no estilo: 

fifo_WIDTH4_BUF_SIZE8

Isso é lógico: uma FIFO de largura 4 e profundidade 8 não é a mesma implementação física que uma FIFO de largura 8 e profundidade 16. Mas para integração isso pode ser chato: o top-level ou o wrapper pode estar instanciando fifo, enquanto a netlist gerada contém outro nome.

Para lembrar: parâmetros não default afetam mais que largura de sinal. Eles podem afetar o nome do módulo elaborado, a lista de portas achatadas e o wrapper necessário para simulação.

Wrapper: a casca que estabiliza o fluxo

O wrapper é um módulo de casca. Ele instancia a interface, instancia o DUT parametrizado e oferece uma fronteira mais controlada para a ferramenta. No fluxo bottom-up, ele ajuda a sintetizar um bloco isolado sem perder a noção de como esse bloco conversa com o restante do sistema. 

module wrapper_fifo #(
  parameter WIDTH    = 8,
  parameter BUF_SIZE = 16
) (
  input  logic clk,
  input  logic rst_n
);
  fifo_io #(WIDTH) fifo_if();

  fifo #(
    .WIDTH(WIDTH),
    .BUF_SIZE(BUF_SIZE)
  ) dut (
    .clk(clk),
    .rst_n(rst_n),
    .fifo_if(fifo_if.fifo)
  );
endmodule

O curso descreve uma sequência mental parecida com esta:

  1. Criar um wrapper SystemVerilog para o design.
  2. Analisar RTL, interface e wrapper.
  3. Elaborar o wrapper com parâmetros desejados.
  4. Definir o design corrente para o bloco alvo real, quando necessário.
  5. Continuar o fluxo de síntese.
  6. Gerar netlist e, se necessário, novo wrapper para simulação gate-level.
analyze -format sverilog { fifo_io.sv fifo.sv wrapper_fifo.sv }
elaborate wrapper_fifo -parameter "WIDTH=4, BUF_SIZE=8"
current_design [get_designs fifo*]
compile
Ligação com o professor: essa é exatamente a zona onde Makefile, scripts de síntese e organização por blocos começam a fazer sentido. O Makefile chama comandos; o wrapper garante uma fronteira previsível para o bloco que está sendo sintetizado/testado.

Por que gerar novo wrapper para gate-level?

Depois da síntese, duas coisas podem ter mudado: a interface foi achatada e o nome do módulo pode ter sido especializado pelos parâmetros. Se você tentar simular a netlist usando o wrapper RTL antigo, ele pode instanciar um nome que não existe ou conectar uma interface que já não existe como objeto.

Por isso a aula mostra a necessidade de gerar código de instanciação ou um novo wrapper compatível com o resultado sintetizado. A simulação gate-level precisa conectar o que a síntese realmente emitiu, não o que era mais bonito no RTL original.

Antes da síntese Depois da síntese Consequência
Porta de interface Portas individuais achatadas Wrapper gate-level deve conectar sinais reais.
Módulo parametrizado genérico Módulo especializado por parâmetros Nome da instância pode precisar ser atualizado.
RTL abstrato e legível Netlist estrutural Simulação exige bibliotecas, delays e conexões corretas.

Aplicação no nosso CPU e no ambiente

Para o nosso projeto, esta aula não significa que precisamos sair colocando interface em tudo imediatamente. A leitura madura é outra: entender quando uma fronteira está ficando complexa o suficiente para merecer uma interface ou um wrapper.

CPU simples Mantenha portas explícitas quando a fronteira ainda é pequena e didática.
Barramento interno Considere interface quando muitos sinais sempre viajam juntos.
Memória ou FIFO Modport ajuda a separar quem é master e quem é slave.
Síntese de bloco Wrapper ajuda a fixar parâmetros e isolar DUT.
Gate-level Espere portas achatadas e nomes possivelmente alterados.
Makefile Pode ter targets separados para analisar, elaborar wrapper, sintetizar e gerar netlist/wrapper final.

Revisão para fixar

1. Qual a diferença entre `struct` e `union`? `struct` coloca campos lado a lado; `union` coloca várias visões sobre o mesmo armazenamento.
2. Como reconhecer packed e unpacked pela posição dos colchetes? Colchetes antes do nome são packed; colchetes depois do nome são unpacked.
3. Por que um unpacked array não recebe simplesmente `8'b1`? Porque ele é uma coleção de elementos, não uma palavra de bits contínua.
4. O que uma interface resolve? Agrupa sinais de comunicação relacionados e reduz portas soltas na fronteira dos módulos.
5. Para que serve modport? Para definir a direção dos sinais da interface conforme o papel do módulo, como master ou slave.
6. O que a síntese faz com interface? Achata a interface em portas individuais e pode renomear sinais.
7. Por que wrappers são úteis? Porque estabilizam a fronteira do bloco, controlam parâmetros e ajudam integração/simulação após síntese.
