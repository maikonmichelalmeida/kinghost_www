FSM, `X`, `case`, arrays e structs em RTL

Esta aula continua a anterior: depois de aprender `enum`, `always_comb`, `always_ff`, `unique` e `priority`, agora vamos ver como essas escolhas afetam máquinas de estado, comparação com wildcards, tri-state e organização de dados.

Objetivo desta aula

A parte B do curso mostra que SystemVerilog tem recursos poderosos, mas cada recurso tem uma intenção. Uma FSM com `enum` pode ficar limpa, mas pode virar uma FSM de 32 bits se você esquecer a largura. Um `case` pode parecer cobrir tudo, mas ainda gerar latch se faltarem defaults. Um `x` pode parecer don't-care, mas em comparação comum ele é unknown. Uma `struct packed` pode ser excelente para barramentos, mas uma fila dinâmica é outro mundo, muito mais voltado a simulação.

A mensagem central é: escreva RTL de modo que a ferramenta não precise adivinhar. Declare largura, declare exclusividade quando ela realmente existir, use operadores de wildcard apropriados e escolha estruturas de dados que façam sentido para hardware sintetizável.

FSM com enum: o detalhe que muda tudo

A aula começa com uma FSM em SystemVerilog usando `enum`. A intenção é boa: nomes de estado deixam a máquina muito mais legível que números mágicos. O problema é que o primeiro exemplo não especifica o tipo base do enum. 

// Cuidado: sem tipo base explícito, o enum pode assumir int
typedef enum {IDLE, INIT, START} state_enum;

state_enum st, nxt;

Se o tipo base assume `int`, a variável de estado pode ter 32 bits. Para três estados, isso é enorme e abre espaço para uma pergunta da síntese: "o que faço com todos os outros valores possíveis que cabem em 32 bits?". Se o `case` só cobre `IDLE`, `INIT` e `START`, não cobre todos os valores possíveis do tipo. Somado a isso, se `nxt` e `dout` não recebem default no `always_comb`, a ferramenta infere latch.

Armadilha do slide: `typedef enum {IDLE, INIT, START} state_enum;` parece pequeno, mas pode não ser. Para RTL, prefira sempre declarar a largura. 
// Forma mais segura para RTL
typedef enum logic [1:0] {
  IDLE,
  INIT,
  START
} state_enum;

state_enum st, nxt;

always_comb begin
  nxt  = st;
  dout = 1'b0;

  unique case (st)
    IDLE:  if (select) nxt = INIT;
    INIT:  begin dout = 1'b1; nxt = START; end
    START: begin dout = 1'b0; nxt = IDLE; end
    default: begin nxt = IDLE; dout = 1'b0; end
  endcase
end

always_ff @(posedge clk or negedge rst_n) begin
  if (!rst_n)
    st <= IDLE;
  else
    st <= nxt;
end

Largura explícita

`enum logic [1:0]` diz à ferramenta que a FSM tem dois bits, não um inteiro genérico.

Default em combinacional

`nxt = st` e `dout = 0` eliminam caminhos sem atribuição.

Default no case

Ajuda a tratar estado inválido de modo previsível, especialmente depois de reset, X ou falha.

Self-check: enum variável não é tipo

O slide mostra algo como:

enum logic [1:0] {IDLE, INIT, START} state_enum;

state_enum st, nxt;

Isso não cria um tipo chamado `state_enum`; cria uma variável chamada `state_enum`. Para criar um tipo e depois declarar variáveis desse tipo, precisa usar `typedef`. 

typedef enum logic [1:0] {
  IDLE,
  INIT,
  START
} state_enum;

state_enum st, nxt;

Binary versus one-hot: não é só gosto

O curso compara estilos de codificação de FSM porque eles influenciam a Quality of Results, ou QoR. QoR é o conjunto de métricas que a síntese entrega: área, timing, número de células, número de nets e, em fluxo mais completo, consumo e congestionamento.

Estilo Ideia Vantagem típica Custo típico
Binary Usa o menor número de bits para codificar os estados. Menos flip-flops. Decodificação combinacional pode ser mais complexa.
One-hot Um bit por estado; em estado válido, só um bit fica em 1. Decodificação simples e, muitas vezes, timing melhor. Mais flip-flops.
One-hot com `case (1'b1)` O `case` testa qual bit do estado está ativo. Expressa diretamente a intenção one-hot. Precisa garantir que a FSM realmente seja one-hot.

Para uma FSM pequena, binary costuma ser natural. Para controladoras com muitos estados e caminhos críticos de decodificação, one-hot pode melhorar timing porque a lógica para saber "estou neste estado?" pode ser apenas ler um bit. Não existe resposta universal: a síntese mede e otimiza, mas o estilo de RTL dá pistas importantes. 

// One-hot com case em 1'b1
always_comb begin
  nxt  = '0;
  dout = 1'b0;

  unique case (1'b1)
    st[0]: if (en) begin
      nxt[1] = 1'b1;
    end else begin
      nxt[0] = 1'b1;
    end

    st[1]: begin
      dout   = ^a;
      nxt[2] = 1'b1;
    end

    st[2]: begin
      dout   = ^(a + b);
      nxt[0] = 1'b1;
    end

    default: nxt[0] = 1'b1;
  endcase
end
Leitura correta: `case (1'b1)` parece estranho no começo. Ele não está comparando estado com um número; está perguntando qual expressão de item vale 1. Em one-hot, cada item costuma ser `st[i]`.

`unique` e `unique0`: exatamente uma ou no máximo uma

A parte A já apresentou `unique`. A parte B acrescenta `unique0`, e essa diferença é uma daquelas que aparecem como pegadinha.

`unique`

Você está dizendo que exatamente uma alternativa válida deve casar. Se nenhuma casar ou se mais de uma casar, há violação da intenção.

`unique0`

Você está dizendo que no máximo uma alternativa deve casar. Nenhuma alternativa casar pode ser permitido.

Em termos de síntese, essas palavras podem ajudar a ferramenta a reconhecer exclusividade e evitar lógica de prioridade desnecessária. Em termos de simulação, elas ajudam a denunciar quando o mundo real do sinal não bateu com a promessa do código.

Regra prática: use `unique` ou `unique0` apenas quando você entende a propriedade que está prometendo. Em FSM one-hot, `unique case (1'b1)` é forte e útil, mas só se o estado realmente tiver um único bit ativo em operação normal.

`X` é unknown, não don't-care

Esta é talvez a parte mais importante do bloco para evitar bugs conceituais. O slide diz: para simulação, `x` é unknown, não don't-care. E reforça que, no enquadramento do curso, em comparação comum para síntese o designer também não deve tratar `x` como wildcard. 

if (In_A[7:0] == 8'b00xx11xx) begin
  Data_Out = 1'b1;
end else begin
  Data_Out = 1'b0;
end

O iniciante olha para `00xx11xx` e pensa: "quero comparar os bits 7:6 e 3:2, ignorando os outros". Mas `==` não faz isso. Ele compara com um padrão que contém desconhecidos. Se o resultado da igualdade é unknown, o `if` não entra no `then`. Em SystemVerilog, no teste de `if`, apenas 1 é verdadeiro; 0, X e Z não são tratados como true.

Errado

Usar `x` dentro de `==` esperando wildcard.

Certo

Usar operador próprio de comparação com máscara: `==?` ou `!=?`.

Mais seguro ainda

Manter `X` como sinal de problema em simulação, não como ferramenta casual de otimização.

`==?` e `!=?`: comparação com máscara

Para dizer "compare só alguns bits e ignore outros", SystemVerilog fornece operadores específicos: ==? e !=?. Eles tratam x, z e ? no operando da direita como wildcards. O lado direito deve ser constante. 

logic [3:0] din;
logic       dout;

assign dout = (din ==? 4'b???0);

// Equivalente lógico:
// dout = ~din[0];

No exemplo, os três bits mais altos do padrão da direita são `?`, então não participam da comparação. O único bit realmente testado é o bit 0, que precisa ser 0.

Detalhe de prova: wildcards valem no operando da direita. Se o sinal do lado esquerdo vier com `X` ou `Z`, isso não vira don't-care automaticamente. 
// Decodificação didática de opcode: compara só campos relevantes
always_comb begin
  is_load_store = 1'b0;

  if (opcode ==? 8'b10??_00??)
    is_load_store = 1'b1;
end

`case inside`, `casez` e `casex`

A aula então separa três ideias que muita gente mistura: `case` comum, `case inside` e os wildcards legados `casez`/`casex`.

Construção Como pensar Cuidado
case Compara itens de forma comum. `?`, `x` e `z` são valores desconhecidos, não wildcard. Não use `3'b0?0` esperando casar com 000 e 010.
case inside Permite expressar pertencimento a conjuntos e intervalos. Ótimo para clareza, mas ainda exige defaults e intenção de síntese clara.
casez Trata `z` e `?` como don't-care nos itens. Mais explícito que `casex`, mas ainda requer cuidado.
casex Trata `x`, `z` e `?` como don't-care nos itens. Pode mascarar `X` real de simulação e esconder bug. 
// case inside: pertencimento a conjunto/faixa
always_comb begin
  action = ACTION_DEFAULT;

  case (status) inside
    3'b000, 3'b001:      action = ACTION_A;
    [3'b010:3'b101]:     action = ACTION_B;
    default:             action = ACTION_C;
  endcase
end
Por que `casex` é tão perigoso?

`casex` pode tratar um `X` vindo do design como don't-care. Só que `X` em simulação muitas vezes é justamente o sinal de que algo está errado: reset incompleto, múltiplos drivers, sinal não inicializado ou caminho sem atribuição. Se você usa `casex`, pode esconder essa evidência e seguir com uma simulação aparentemente limpa.

Em RTL moderno, prefira `case inside`, `==?`, `unique case`, `priority case` ou codificação explícita. Use `casex` apenas se houver motivo muito claro e revisão cuidadosa.

Tri-state: regra de síntese

O slide de tri-state é curto, mas importante. Para sintetizar tri-state conforme mostrado no curso, o sinal deve ser uma net, como wire, e deve ser dirigido por atribuição contínua. O ramo com 'z expressa alta impedância. 

wire D_Out;

assign D_Out = (Use_B) ? B : 'z;

Isso significa: quando Use_B vale 1, o sinal dirige B. Quando vale 0, ele não dirige o barramento, deixando alta impedância. Em ASIC moderno, tri-state interno muitas vezes é evitado ou convertido em mux. Ainda assim, para pinos de I/O ou barramentos específicos, a regra da aula é: tri-state sintetizável precisa de net e continuous assignment.

Para o CPU: dentro do datapath, normalmente pense em mux, não em tri-state. Se aparecer `'z` em lógica interna, trate como ponto de revisão antes de síntese.

Arrays packed e unpacked

SystemVerilog separa duas dimensões que o Verilog antigo deixava menos expressivas: dimensões packed e unpacked. Isso é fundamental para modelar memórias, barramentos e matrizes de registros sem perder o mapa mental.

Packed

Fica antes do nome. Representa bits agrupados como um vetor contínuo. É natural para barramentos e campos bit-a-bit. 

logic [7:0] byte_value;

Unpacked

Fica depois do nome. Representa coleção de elementos. É natural para memórias, bancos de registradores e tabelas fixas. 

logic [7:0] memory [0:127];

No exemplo logic [7:0] memory [0:127];, cada elemento tem 8 bits packed e existem 128 elementos unpacked. Isso conversa diretamente com banco de registradores e memória do CPU. 

// Banco de 32 registradores, cada um com 32 bits
logic [31:0] regfile [0:31];

assign read_data = regfile[read_addr];
Dynamic arrays, queues e associative arrays

O curso cita arrays dinâmicos, filas e arrays associativos, mas o ponto prático para RTL é separar linguagem de design sintetizável de linguagem de testbench. Recursos dinâmicos são muito úteis em simulação, scoreboards, filas de transações e modelos de referência. Para RTL sintetizável comum, prefira estruturas de tamanho fixo.

Struct packed versus unpacked

`struct` ajuda a dar nome a grupos de sinais. Em vez de passar vários sinais soltos, você pode agrupar campos que pertencem ao mesmo conceito: uma instrução decodificada, uma transação de barramento, um pacote de controle.

Tipo Como pensar Uso natural
struct packed Um vetor de bits com campos nomeados. Barramentos, instruções, pacotes de controle, campos que precisam virar bits contínuos.
struct unpacked Uma coleção de membros separados. Organização lógica de variáveis, especialmente quando não precisa tratar como vetor único. 
typedef struct packed {
  logic        valid;
  logic [4:0]  rd;
  logic [2:0]  alu_op;
  logic        mem_read;
  logic        mem_write;
} control_t;

control_t ctrl;

always_comb begin
  ctrl = '0;

  if (is_add) begin
    ctrl.valid  = 1'b1;
    ctrl.rd     = instr[11:7];
    ctrl.alu_op = ALU_ADD;
  end
end

Repare como isso pode ajudar no nosso CPU: sinais de controle deixam de ficar espalhados como fios soltos e passam a formar um grupo com nome. Isso não deve virar abstração exagerada, mas pode deixar decode e controle muito mais revisáveis.

Quando `struct packed` é melhor que vários sinais?

Use quando os campos viajam juntos, são atribuídos juntos ou representam uma unidade conceitual. Por exemplo: controle de uma instrução decodificada, resposta de uma ALU ou interface interna simples. Evite quando o agrupamento só torna conexões mais difíceis sem melhorar entendimento.

Union: múltiplas visões do mesmo armazenamento

O bloco termina começando o assunto de `union`. A ideia é parecida com C: permitir que o mesmo espaço seja visto por formatos diferentes. 

typedef union {
  tcp_t          tcp_h;
  udp_t          udp_h;
  logic [63:0]   bits;
  logic [7:0][7:0] bytes;
} union_unpacked_t;

Em design digital, isso é útil quando o mesmo conjunto de bits pode ser interpretado por campos diferentes. Mas esta parte só abre o tema; a continuação entra no próximo bloco. Para o estudo atual, basta guardar a ideia: `struct` agrupa campos; `union` permite visões alternativas do mesmo armazenamento.

Aplicação no nosso projeto

Esta aula dá um conjunto de revisões bem concreto para o CPU e para qualquer bloco que vamos encaixar no ambiente automatizado do professor.

FSMs Declarar enum com largura explícita e usar defaults em `always_comb`.
Estados inválidos Ter `default` ou estratégia clara para recuperar para estado seguro.
QoR Ver se binary ou one-hot faz mais sentido depois de olhar reports de área/timing.
Wildcards Não usar `x` em `==` como máscara; usar `==?` quando a máscara for intencional.
Case Evitar `casex` como hábito; preferir formas explícitas.
Dados Usar arrays fixos e structs quando eles deixarem o RTL mais claro e sintetizável.

Revisão para fixar

1. O que acontece se `enum` não tiver tipo base explícito? Pode assumir `int`, frequentemente 32 bits, criando uma FSM maior que o necessário e dificultando cobertura total dos valores possíveis.
2. Por que `nxt` sem default em `always_comb` pode gerar latch? Porque há caminhos em que `nxt` não recebe novo valor; para preservar valor anterior, a síntese precisa memória.
3. Qual a diferença entre `unique` e `unique0`? `unique` espera exatamente uma correspondência válida. `unique0` espera no máximo uma correspondência válida.
4. `x` em `==` significa don't-care? Não. Em comparação comum, `x` é unknown. Para máscara, use `==?` ou `!=?`.
5. Onde os wildcards de `==?` valem? No operando da direita, que deve ser constante. `x` ou `z` no lado esquerdo não viram wildcard.
6. Qual a diferença entre packed e unpacked? Packed é vetor de bits contínuo; unpacked é coleção de elementos.
7. Quando usar `struct packed`? Quando a estrutura representa bits físicos agrupados, como campos de controle, instrução ou barramento.
