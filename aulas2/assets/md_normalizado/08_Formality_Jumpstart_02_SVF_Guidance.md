## Resumo executivo

Esta aula apresenta o papel do **SVF** dentro do fluxo de verificação de equivalência com o **Formality**. O ponto central é que, quando o **Design Compiler** sintetiza um RTL para uma netlist otimizada, ele pode realizar transformações que tornam a comparação direta entre RTL e netlist muito difícil: mudança de nomes, remoção de registradores constantes, merge/duplicação de registradores, re-encoding de FSM, retiming, inversão de fase de registradores e mudanças em datapath.

O **SVF**, chamado no slide de **Automated Guidance Setup file**, funciona como um arquivo de orientação. Ele leva ao Formality informações sobre o que o Design Compiler fez durante a síntese. Isso reduz setup manual, evita erros de configuração e diminui iterações desnecessárias de verificação.

A aula também introduz o **Auto Setup Mode**, controlado pela variável `synopsys_auto_setup`. Esse modo faz o Formality assumir várias convenções semelhantes às usadas pelo Design Compiler, aumentando a chance de sucesso inicial da verificação. O cuidado importante é que ele deve ser usado especialmente em verificações **RTL-to-gate**, e não em verificações **gate-to-gate**.

Por fim, o bloco mostra o comando `fm_mk_script`, que gera automaticamente um script Tcl de Formality a partir de um SVF. Isso é útil para criar rapidamente uma base de script, especialmente em fluxos Synopsys integrados.

## Aula didática desenvolvida

### 1. Por que o SVF existe?

Na aula anterior, o Formality foi apresentado como uma ferramenta de **equivalence checking**. A ideia básica é comparar dois designs:

- o **reference design**, normalmente o RTL considerado correto;
- o **implementation design**, normalmente a netlist produzida pela síntese.

Em um caso simples, a comparação seria direta. O RTL tem uma saída `sum`, a netlist tem uma saída `sum`. O RTL tem um registrador `a_reg`, a netlist tem um registrador equivalente. O Formality encontra os compare points, casa os nomes e prova se os cones de lógica são equivalentes.

Mas síntese real raramente é tão simples. O **Design Compiler** não apenas traduz RTL para portas; ele otimiza. E essas otimizações podem alterar profundamente a estrutura aparente do design, mesmo preservando a função.

Exemplos:

- um registrador pode ser removido porque sempre recebe valor constante;
- dois registradores equivalentes podem ser fundidos;
- um registrador pode ser duplicado para melhorar fanout;
- a codificação de uma FSM pode ser alterada;
- registradores podem ser deslocados por retiming;
- a polaridade de um registrador pode ser invertida;
- multiplicadores e divisores podem ser implementados com arquiteturas diferentes;
- um datapath pode ser reestruturado.

Do ponto de vista funcional, o design pode continuar correto. Mas, do ponto de vista estrutural, ele ficou diferente. O SVF existe para contar ao Formality: “durante a síntese, estas transformações foram feitas; use essas informações para guiar a equivalência”.

---

### 2. O SVF não é uma prova: ele é uma orientação

Uma ideia delicada do slide é que o SVF carrega **hints**, ou seja, dicas/orientações. Isso é diferente de dizer que o SVF “manda” o Formality aceitar tudo.

O slide deixa claro:

```text
SVF data is implicitly or explicitly proven in Formality or it is not used
```

Isso significa que o Formality usa a orientação do SVF dentro de um processo formal. Ele não simplesmente acredita no arquivo. Se a informação puder ser validada no contexto da comparação, ela ajuda a formar os mapeamentos. Se não puder, ela não deve ser usada.

Essa distinção é crucial:

- **Design Compiler** informa a transformação.
- **Formality** usa a informação para orientar match/verify.
- **A equivalência ainda precisa ser provada.**

Em termos práticos, o SVF melhora a capacidade do Formality de encontrar correspondências corretas, mas não substitui a verificação.

### 3. Fluxo mental correto: Design Compiler gera, Formality lê

O fluxo sugerido pela aula é:

1. No Design Compiler, habilitar a geração do SVF.

```tcl
# configuração, match e verify.
```

O ponto importante é que `synopsys_auto_setup` deve ser definido antes de `set_svf`.

No **Formality**, esse comando carrega o arquivo para orientar a verificação.


O arquivo pode ter nome padrão:


ou nome definido pelo usuário:

```tcl

```

### 5. Auto Setup Mode: reduzir diferenças artificiais entre síntese e Formality

O Auto Setup Mode procura fazer o Formality interpretar certas situações do mesmo modo que o Design Compiler interpretou durante a síntese.

Isso é importante porque, em RTL, certas construções podem ser ambíguas ou dependentes de diretivas. Por exemplo:

- sinais não dirigidos;
- `full_case`;
- `parallel_case`;
- tratamento de scan;
- clock gating;
- interpretação de certas informações de FSM;
- informações de UPF em fluxos low power.

Se o Design Compiler e o Formality interpretarem essas situações de modos diferentes, o resultado pode ser uma falha falsa.

Exemplo conceitual:

```verilog
always_comb begin
  case (sel)
    2'b00: y = a;
    2'b01: y = b;
  endcase
end
```

Se o RTL não define explicitamente o que acontece para `sel = 2'b10` ou `sel = 2'b11`, a interpretação pode depender de como o código foi escrito, de pragmas e de suposições de síntese. Se a síntese assumiu uma condição e a verificação formal assumiu outra, haverá divergência. O problema não é necessariamente uma netlist errada; o problema pode ser diferença de setup.

O Auto Setup Mode tenta alinhar essas interpretações.

---

### 6. Por que não usar Auto Setup Mode em gate-to-gate?

O slide afirma explicitamente:

```text
You should not use auto setup mode for gate-to-gate verifications
```

Em uma comparação **RTL-to-gate**, existe uma distância semântica maior: um lado é RTL, o outro é netlist. Então faz sentido ajudar o Formality a interpretar o RTL de forma compatível com a síntese.

Em uma comparação **gate-to-gate**, os dois lados já são netlists. Não há a mesma necessidade de aplicar pressupostos de interpretação de RTL. Usar auto setup pode introduzir configurações desnecessárias ou perigosas, porque a comparação deve ser mais estruturalmente controlada.

Portanto:

```text
RTL-to-gate: auto setup costuma ajudar.
Gate-to-gate: não usar auto setup como padrão.
```

## Conceitos difíceis explicados em profundidade

### 1. SVF — Automated Guidance Setup file

**SVF** é o arquivo de guidance gerado pelo Design Compiler para informar ao Formality como a síntese transformou o design.

Ele contém dois tipos de informação:

1. **Setup information**: variáveis, opções e configurações que ajudam o Formality a montar um ambiente de verificação compatível com a síntese.
2. **Guidance information**: informações de mapeamento e transformação, como renomeações, merges, duplicações, re-encoding e retiming.

O SVF é especialmente importante quando a implementação não se parece estruturalmente com o RTL original.

---

### 2. Object name changes

Durante a síntese, nomes podem mudar. Um registrador RTL como:

```text
a_reg[31]
```

pode aparecer na netlist como algo próximo de:

```text
a_reg_31_
```

ou com um nome ainda mais transformado.

Sem guidance, o Formality pode ter dificuldade para casar os compare points automaticamente. O SVF informa essas mudanças para facilitar o match.

---

### 3. Constant register optimizations

Se um registrador sempre recebe `0` ou `1`, a síntese pode removê-lo e substituí-lo por uma constante.

Exemplo conceitual:

```verilog
always_ff @(posedge clk) begin
  mode_reg <= 1'b0;
end
```

A síntese pode concluir que `mode_reg` não precisa existir fisicamente. Na netlist, esse ponto pode virar uma conexão fixa em `0`.

Para equivalência, o Formality precisa entender que o registrador do RTL foi otimizado para uma constante. O SVF carrega essa informação.

---

### 4. Duplicate and merged registers

A síntese pode duplicar registradores para melhorar timing/fanout.

Exemplo conceitual:

```text
RTL:
  ctrl_reg alimenta 200 cargas

Netlist:
  ctrl_reg_replica1 alimenta parte das cargas
  ctrl_reg_replica2 alimenta outra parte das cargas
```

Funcionalmente, as réplicas devem carregar o mesmo valor lógico, mas estruturalmente agora existem mais registradores na implementação do que no RTL.

O contrário também pode acontecer: dois registradores equivalentes no RTL podem ser fundidos em um só registrador na netlist.

Sem guidance, isso cria problemas de match:

```text
1 registrador no reference -> 2 registradores no implementation
2 registradores no reference -> 1 registrador no implementation
```

O SVF informa essas relações.

---

### 5. Multiplier and divider architecture types

Multiplicadores e divisores podem ser sintetizados com arquiteturas muito diferentes.

Por exemplo, um multiplicador pode ser implementado como:

- array multiplier;
- Booth multiplier;
- Wallace tree;
- estrutura baseada em DesignWare;
- versão otimizada para área;
- versão otimizada para timing.

O RTL pode dizer apenas:

```verilog
assign y = a * b;
```

Mas a netlist pode conter uma arquitetura complexa. O SVF ajuda o Formality a reconhecer que aquela implementação corresponde ao operador aritmético original.

---

### 6. Datapath transformations

Datapaths são blocos de operação sobre dados: somadores, subtratores, comparadores, multiplicadores, muxes, acumuladores etc.

A síntese pode reordenar operações usando propriedades algébricas.

Exemplo conceitual:

```verilog
assign y = (a + b) + c;
```

pode ser reestruturado como:

```verilog
assign y = a + (b + c);
```

Em aritmética binária com certas condições, essas transformações podem preservar a função, mas mudar a estrutura dos cones lógicos.

O SVF ajuda o Formality a entender essas transformações de datapath.

---

### 7. FSM re-encoding

Uma FSM pode ser escrita no RTL com uma codificação e sintetizada com outra.

Exemplo:

```verilog
typedef enum logic [1:0] {
  IDLE = 2'b00,
  READ = 2'b01,
  EXEC = 2'b10,
  DONE = 2'b11
} state_t;
```

A síntese pode trocar para one-hot:

```text
IDLE = 0001
READ = 0010
EXEC = 0100
DONE = 1000
```

Nesse caso, os bits de estado não são equivalentes bit a bit. O estado lógico é equivalente, mas a codificação mudou.

Sem SVF, o Formality poderia comparar os registradores de estado diretamente e concluir que são diferentes. Com SVF, ele recebe a informação de re-encoding e pode comparar a máquina em nível de comportamento equivalente.

---

### 8. Retiming

Retiming é uma transformação sequencial em que registradores são movidos através de lógica combinacional para melhorar timing.

Exemplo conceitual:

```text
Antes:
  Reg A -> lógica combinacional grande -> Reg B

Depois:
  Reg A -> parte da lógica -> Reg novo/reposicionado -> resto da lógica -> Reg B
```

O comportamento observado nas fronteiras pode continuar equivalente, mas os registradores internos deixam de estar no mesmo lugar.

Essa é uma das situações em que o slide diz que o SVF é **required**. A razão é simples: o match registro-a-registro deixa de funcionar de forma direta.

---

### 9. Register phase inversion

A síntese pode inverter a fase de um registrador e compensar essa inversão na lógica ao redor.

Exemplo conceitual:

```text
Reference:
  q = valor

Implementation:
  q_inv = ~valor
  lógica seguinte compensa a inversão
```

Se o Formality comparar `q` com `q_inv` diretamente, parecerá falha. Mas, considerando a inversão de fase, a lógica pode ser equivalente.

O SVF informa esse tipo de transformação.

---

### 10. Undriven signals

Sinais não dirigidos são sinais que não recebem valor claro no design.

Dependendo da ferramenta, um sinal não dirigido pode ser tratado como:

- `X`;
- `0`;
- `1`;
- don't care;
- valor compatível com síntese.

A variável mostrada no slide:

```tcl
set verification_set_undriven_signals synthesis
```

orienta o Formality a tratar esses sinais como a síntese trataria. Isso reduz divergências artificiais.

---

### 11. `full_case` e `parallel_case`

Os comandos do slide incluem:

```tcl
set hdlin_ignore_parallel_case false
set hdlin_ignore_full_case false
```

Como o nome da variável contém `ignore`, o valor `false` significa que o Formality **não deve ignorar** essas diretivas.

- `full_case` informa à síntese/verificação que todos os casos relevantes foram cobertos, mesmo que o código não liste todas as combinações.
- `parallel_case` informa que as condições são mutuamente exclusivas, permitindo otimizações.

Essas diretivas são perigosas se usadas de forma incorreta, porque podem fazer a síntese assumir algo que a simulação RTL não revela diretamente. O Auto Setup Mode ajuda o Formality a seguir a mesma interpretação usada na síntese.

---

### 12. Clock-gating notification

Clock gating é uma técnica de redução de potência em que o clock de um bloco é desligado quando o bloco não precisa atualizar estado.

A síntese pode inserir clock gating automaticamente. Isso altera a estrutura do circuito, introduzindo células de gating no caminho de clock.

O slide indica que o Design Compiler pode colocar informações de clock-gating no SVF. Isso ajuda o Formality a entender que a presença de lógica de clock gating na implementação é uma transformação esperada, não necessariamente uma diferença funcional.

---

### 13. Scan mode disable information

Scan é uma estrutura DFT usada para teste de fabricação. Durante verificação funcional de equivalência, normalmente não se quer comparar o comportamento em modo scan como se fosse comportamento funcional normal.

O slide indica que o Design Compiler coloca no SVF informações para desabilitar scan mode quando apropriado.

Isso evita que a lógica de teste interfira na comparação funcional entre reference e implementation.

---

### 14. `synopsys_auto_setup_filter`

O slide menciona:

```tcl
synopsys_auto_setup_filter
```

Essa variável permite controlar seletivamente o que o auto setup deve aplicar. Isso é útil quando o auto setup resolve a maior parte do problema, mas alguma configuração específica precisa ser excluída ou tratada manualmente.

A ideia prática é:

```text
Use auto setup como ponto de partida, mas mantenha controle fino quando necessário.
```

---

### 15. `fm_mk_script`

O comando:

```tcl
fm_mk_script <svf> [-o[utput] <file>]
```

gera um script de Formality usando informações do SVF.

Ele é especialmente útil quando:

- você tem um SVF produzido por síntese;
- quer gerar rapidamente um script inicial de verificação;
- deseja reduzir erro manual;
- precisa entender quais comandos básicos o fluxo espera.

Exemplos:

```tcl
fm_mk_script default.svf
```

gera:

```text
fm_mk_script.tcl
```

E:

```tcl
fm_mk_script default.svf -output fm.tcl
```

gera:

```text
fm.tcl
```

## Pontos de prova e revisão

### Perguntas prováveis

1. **O que é SVF?**  
   É um Automated Guidance Setup file, gerado pelo Design Compiler, que passa informações de setup e guidance ao Formality.

2. **Quem gera o SVF?**  
   O Design Compiler.

3. **Quem lê o SVF?**  
   O Formality.

4. **Qual é o nome padrão do SVF?**  
   `default.svf`.

5. **Qual comando especifica o SVF?**  
   `set_svf file.svf`.

6. **O SVF é apenas um arquivo de setup?**  
   Não. Ele contém setup e guidance information.

7. **O Formality confia cegamente no SVF?**  
   Não. O dado do SVF é provado implicitamente ou explicitamente no Formality, ou não é usado.

8. **Quando o fluxo SVF é requerido segundo o slide?**  
   Quando a netlist contém retiming, register merging ou register inversions.

9. **Quais transformações podem aparecer no SVF?**  
   Mudanças de nomes, otimizações de registradores constantes, registradores duplicados/fundidos, arquitetura de multiplicadores/divisores, transformações de datapath, FSM re-encoding, retiming e register phase inversion.

10. **Qual variável ativa o Auto Setup Mode?**  
    `synopsys_auto_setup`.

11. **Qual comando típico ativa o Auto Setup Mode?**  
    `set synopsys_auto_setup true`.

12. **A variável `synopsys_auto_setup` deve ser definida antes ou depois do `set_svf`?**  
    Antes.

13. **Auto setup funciona sem SVF?**  
    Sim, mas faz mais com SVF.

14. **O que o Auto Setup Mode busca alinhar?**  
    As suposições feitas no Design Compiler com as suposições feitas no Formality.

15. **Auto Setup Mode deve ser usado em gate-to-gate verification?**  
    Não, segundo o slide.

16. **O que são false failures?**  
    Falhas aparentes de verificação causadas por setup incorreto ou ausente, não por diferença funcional real.

17. **Qual comando gera automaticamente um script Tcl de Formality a partir de SVF?**  
    `fm_mk_script`.

18. **Qual é o nome padrão do script gerado por `fm_mk_script default.svf`?**  
    `fm_mk_script.tcl`.

19. **Como gerar um script chamado `fm.tcl`?**  
    `fm_mk_script default.svf -output fm.tcl`.

---

### Pegadinhas

#### Pegadinha 1 — “SVF é prova de equivalência”

Errado. O SVF é guidance. A prova é feita pelo Formality.

#### Pegadinha 2 — “Auto setup sempre deve ser usado”

Errado. O slide diz explicitamente que não se deve usar auto setup mode em verificações gate-to-gate.

#### Pegadinha 3 — “`set_svf` é só comando do Formality”

Errado. O slide mostra que o comando é usado no Design Compiler para especificar o arquivo e no Formality para lê-lo.

#### Pegadinha 4 — “Se `hdlin_ignore_full_case` está em `false`, então full_case é ignorado”

Cuidado com o nome da variável. Ela se chama `ignore`. Se `ignore_full_case` é `false`, então o Formality **não ignora** `full_case`.

#### Pegadinha 5 — “FSM re-encoding sempre permite comparar bits de estado diretamente”

Errado. Quando a codificação muda, os bits podem não ser equivalentes bit a bit. A equivalência é de comportamento, não necessariamente de codificação binária.

#### Pegadinha 6 — “Retiming mantém registradores no mesmo lugar”

Errado. Retiming move registradores através da lógica combinacional. Por isso, o SVF é especialmente importante.

## Relação com projeto/laboratório

Este bloco é diretamente útil para qualquer laboratório ou projeto que envolva:

- Design Compiler;
- Formality;
- equivalence checking;
- scripts Tcl;

```tcl
# Possível início de script no Formality
```

Depois entram os comandos de leitura do reference e implementation, por exemplo:

```tcl
read_verilog -r rtl_design.v
```

A sintaxe exata pode variar conforme o material do lab, mas a ideia central será:

```text
-r = reference
-i = implementation
```

O SVF será especialmente importante se a netlist tiver passado por otimizações agressivas. Se houver falha no Formality, uma das primeiras perguntas de debug será:

```text
O SVF correto foi carregado?
O auto setup foi ativado antes do set_svf?
O arquivo SVF corresponde exatamente à netlist sintetizada?
O Design Compiler gerou o SVF durante a síntese correta?
```

Outra boa prática é verificar se a versão textual foi gerada:

```text
formality_svf/svf.txt
```

Ela pode ajudar a entender quais orientações chegaram ao Formality.
