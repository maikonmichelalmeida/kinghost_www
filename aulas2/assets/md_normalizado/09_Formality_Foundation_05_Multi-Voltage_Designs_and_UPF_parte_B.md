## Resumo executivo

Esta parte B é a parte mais prática da aula. A parte A explicou os fundamentos de **UPF**, **power domains**, **isolation cells**, **UPF/UPF’/UPF’’**, scripts com `load_upf`, `verification_force_upf_supplies_on` e `analyze_upf`. Agora a aula mostra como depurar uma falha de equivalência quando a causa pode estar ligada ao power intent.

O sintoma mais importante em designs UPF é a presença de `X`. Em uma verificação tradicional, uma diferença `1/0` costuma indicar diferença funcional direta. Em UPF, uma diferença envolvendo `X` frequentemente aponta para domínio desligado, propagação indevida de power-down, isolation mal controlada, supply net super-restrita, cutpoint em fronteira de domínio ou modelo de tech cell com função de power-down incorreta.

A sequência de debug apresentada é:

1. Ler as mensagens de `load_upf`.
2. Observar se os failing patterns mostram diferença `X` ou diferença `1/0`.
3. Usar a Pattern Window para ver supplies, controles e `X`.
4. Usar a Matched Point Window para localizar `X` em pinos hierárquicos.
5. Abrir o Cone Schematic.
6. Colorir por Power Domain e agrupar por Parent.
7. Usar `Ungroup Selected` para entrar no bloco suspeito.
8. Usar **Find X Sources** para localizar a origem do `X`.
9. Entender `PDCut`/`CutPins` em fronteiras de power domains.
10. Esconder supply nets quando elas poluem o cone.
11. Abrir tech cells para checar power-down functions.
12. Usar `report_upf` para confirmar o que foi inferido a partir do UPF.
13. Conferir assumptions de portas de topo.

- Gates pós-DC devem usar UPF’ gerado pelo Design Compiler.
- Gates pós-ICC devem usar UPF’’ gerado pelo IC Compiler.

### 2. Failure Symptoms

Texto do slide:

- **Failing patterns show X differences at compare point**
  - CutPin
  - Primary output
  - Black box inputs
  - Registers
  - Something in the netlist is powered off causing X to propagate
- **Failing patterns show 1/0 differences at the compare point**
  - maybe not a power-off related problem

Interpretação:

A primeira triagem é olhar se a falha é `X` ou `1/0`.

```text
Diferença com X  → suspeitar de UPF, power-down, isolation, supply ou cutpoint.
Diferença 1/0    → pode ser diferença funcional comum.
```

Isso não substitui a análise, mas orienta o caminho. Se aparece `X`, antes de procurar “a porta lógica errada”, olhe domínio de potência, estado das supplies, controle de isolamento e propagação de power-down.

### 3. Pattern Window — UPF cones are unusual

A Pattern Window mostrada no slide inclui sinais de supply, como:

```text
VSS
VDD
VDDadd
```

e sinais de controle como:

```text
AC
AS
SL
SO
CLK
```

Além dos sinais lógicos normais, aparecem valores `0`, `1` e `X`.

Em UPF, a Pattern Window é mais rica e mais confusa do que em um exemplo RTL simples. O valor de um compare point pode depender de:

- estado ON/OFF da supply;
- power state table;
- controle de isolation;
- controle de retention;
- função de power-down da célula;
- domínio de potência do bloco;
- pinos de power e ground conectados pelo UPF.

Por isso o slide diz que **UPF cones are unusual**. A análise não é só booleana; ela também envolve estado de potência.

### 8. Finding Sources of X — origem localizada

O slide mostra um elemento como:

```text
add1/sum[7]_UPF_ISO/IN_pd
```

e uma anotação:

```text
Don't Care: OUT = ...
Power Domain: PD_adder1
```

Interpretação:

O `X` está associado a um power domain específico:

```text
PD_adder1
```

Isso significa que a origem da falha provavelmente não é uma porta lógica comum. A diferença está ligada a um comportamento de power-down ou a uma célula low-power.

A informação “Don't Care” indica que, naquele estado de potência, a saída pode ser tratada como desconhecida ou irrelevante, dependendo da semântica UPF.

### 9. Separate Power and ISO Controls — X propagates through ISO cell

O slide mostra uma célula de isolamento com pinos de:

```text
POWER
ISO
IN
GROUND
OUT
```

e a frase:

```text
X propagates through ISO cell
```

Interpretação:

Uma isolation cell só protege corretamente se o controle de isolamento estiver ativo no momento correto. A sequência esperada é:

```text
assert ISO antes do domínio de origem desligar
depois desligar o domínio de origem
```

Se power e ISO são controlados separadamente e a sequência está errada, pode haver uma janela em que o domínio já está desligado, mas a isolation cell ainda não está fazendo clamp. Nesse caso, o `X` atravessa para o domínio ligado.

Esse é um caso típico de problema de protocolo de power-down.

```text
If you see an X difference at a compare point when you have set

```

possíveis causas:

- You probably have a model with a bad power down function.
  - Nothing should be off.
- Or something is undriven.

Interpretação:

Este é um ponto de prova importante.

Se:


então todas as supplies estão forçadas ON. Nesse modo, nada deveria estar desligado. Portanto, se mesmo assim aparece `X`, o problema não deve ser simplesmente “domínio desligado”.

- modelo de célula com função de power-down errada;
- sinal sem driver;
- UPF carregado incorretamente;
- supply não conectada como esperado;
- tech cell modelando `X` indevidamente.

### 19. Top Level Port Assumptions

Texto extraído:

- By default Design Compiler assumes that when you do `compile_ultra` the top level ports are driven and read by the top level domain UPF supply.
- The following UPF command will make that assumption explicit for all tools in the flow:

```tcl
set_port_attribute -elements {.} -attribute     related_supply_default_primary true
```

Interpretação:

O Design Compiler assume que as portas de topo usam a supply primária do domínio de topo. Mas deixar isso implícito pode causar diferenças de interpretação entre ferramentas.

O comando acima torna a assumption explícita para o fluxo inteiro. Isso reduz risco de uma ferramenta interpretar a porta de topo de um jeito e outra ferramenta interpretar diferente.

## Aula didática desenvolvida

### 1. Como pensar uma falha UPF

Em um design sem UPF, uma falha geralmente significa que duas funções booleanas são diferentes. Em UPF, isso ainda pode acontecer, mas há uma nova classe de falhas: diferenças causadas por estados de potência.

A pergunta inicial deve ser:

```text
A falha é uma diferença lógica 1/0 ou envolve X?
```

Se envolve `X`, investigue power intent antes de concluir que a lógica funcional está errada.

---

### 2. `X` em UPF não é sempre o mesmo tipo de `X`

Um `X` pode vir de:

- domínio desligado;
- célula de power-down;
- falta de isolation;
- isolation control atrasado;
- supply OFF;
- supply super-restrita;
- tech cell mal modelada;
- sinal sem driver.

A origem muda completamente o diagnóstico. Por isso a aula insiste em Pattern Window, Find X Sources e Cone Schematic.

---

### 3. Debug visual precisa de organização

Cones UPF podem ser muito difíceis de ler. Antes de interpretar, ajuste a visualização:

```text
Cell Coloring: Power Domain
Group All by Parent
Hide Supply Nets se necessário
Ungroup Selected no bloco suspeito
```

Isso transforma um grafo caótico em uma análise por blocos e domínios.

---

### 4. Find X Sources é o equivalente UPF de procurar a causa raiz

Em falhas comuns, você procura a porta ou expressão que divergiu. Em UPF, você procura a origem do `X`.

O botão **Find X Sources** ajuda a descobrir se o `X` vem de:

- célula ISO;
- power-down input;
- supply;
- domínio desligado;
- tech cell.

---

### 5. CutPins indicam fronteira de potência

Se o relatório mostra `CutPin` ou `PDCut`, o problema está associado a uma fronteira de power domain. Isso muda a leitura da falha.

Não trate `PDCut` como um flop comum. Leia as mensagens ATTENTION e investigue a fronteira entre domínios.

---

### 6. Supplies ON por default muda a interpretação do `X`

Com:

```tcl
verification_force_upf_supplies_on true
```

o Formality força supplies ON. Portanto, se aparece `X` mesmo assim, algo está errado:

- modelo de power-down ruim;
- sinal undriven;
- problema de biblioteca;
- problema de UPF.

Isso é pegadinha de prova: o default ajuda debug inicial, mas não é signoff completo.

---

### 7. `report_upf` fecha a investigação textual

Depois da análise gráfica, `report_upf` ajuda a confirmar quantitativamente:

```text
quantas isolation cells?
quantas retention cells?
quantos switches?
qual UPF foi carregado?
qual container?
existe PST?
```

Se o relatório não bate com o esperado, o problema pode estar no arquivo UPF carregado ou na forma como foi carregado.
